## code rule

### 1. 编码前思考

不假设、不隐藏混乱、明确权衡。

LLM 会默默选择一种解释然后推进。这个原则强制显式推理：

- 明确假设 — 如果不确定，不要猜测，而是提问
- 呈现多种解释 — 存在歧义时，不默默选择
- 必要时反驳 — 如果有更简单的方法，就提出来
- 混乱时就停下来 — 明确指出哪里不清楚，并提出问题

### 2. 简单至上

解决问题所需的最少代码。不做投机性实现。

针对过度工程化倾向的对策：

- 不实现超出要求的功能
- 不为只使用一次的代码做抽象
- 不要没被要求的“灵活性”或“可配置性”
- 不为不太可能发生的场景做错误处理
- 如果 200 行能用 50 行写完，就重写

**测试：** 资深工程师会说“过度复杂”吗？如果是，就简化。

### 3. 外科手术式修改

只触碰必须触碰的部分。只清理自己制造的混乱。

修改现有代码时：

- 不要“改善”相邻的代码、注释或格式
- 不要重构没有坏的部分
- 即使自己会写得不一样，也要遵循现有风格
- 如果发现了无关的死代码，不要删除，只提及即可

当自己的修改产生了孤立代码（孤儿）时：

- 因自己的修改而不再需要的 import/变量/函数，要删除
- 之前就存在的死代码，除非被要求，否则不删除

**测试：** 你所修改的每一行，是否都直接关联到用户的要求？

### 4. 目标驱动执行

定义成功标准，循环直到通过验证。

将指令式任务转换为可验证的目标：

| 不要这样 | 而是这样转换 |
|---------|------------|
| “添加验证” | “编写针对无效输入的测试，然后让测试通过” |
| “修复 bug” | “编写能复现该 bug 的测试，然后让测试通过” |
| “重构 X” | “确保重构前后测试都能通过” |

对于多步骤任务，明确写出简洁的计划：

```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```
