# P4.5 Enterprise Observability And Release Readiness Task

优先级：企业级 P4

## 任务定位

本任务用于建立企业级观测、审计、决策回放和上线准入能力。

当前项目已经有结构化日志、request_id、trace_id、graph_path、tool execution log、approval store、checkpoint snapshot 等基础。但企业级 Agent 系统不仅要能跑，还要回答：

- 这次为什么走了这个 intent？
- 为什么选择这个 agent/skill？
- LLM 有没有失败或 fallback？
- 工具有没有失败、超时或被审批拦截？
- 最终回答为什么被 verification patch/block？
- 上线前是否确认没有 mock、eval 是否通过、工具契约是否完整？

本任务目标是让失败请求可定位、关键指标可统计、发布前有准入检查。

## 当前问题

### 1. 日志有事件，但缺少统一 span 模型

当前 `log_event` 已经记录很多事件，但还不是统一 span：

- request span
- graph node span
- llm span
- tool span
- verification span

缺少统一字段和耗时聚合。

### 2. 缺少关键质量指标

企业级 Agent 需要关注：

- rewrite fallback rate
- intent unknown rate
- agent selection clarification rate
- skill no-match rate
- tool failure rate
- verification block rate
- approval resume success rate

当前还没有系统性统计。

### 3. 缺少 request_id 决策回放

单次请求排查时，需要通过 request_id 看到：

- 原始 query
- rewritten query
- intent/sub_intent
- selected agent
- selected skill
- fallback reason
- LLM prompt version
- tool calls
- verification result
- final answer

当前信息分散在多个存储中。

### 4. 缺少上线准入脚本

上环境前需要自动检查：

- 是否禁 mock
- taxonomy/card/skill coverage 是否通过
- prompt manifest 是否完整
- eval 是否通过
- tool contracts 是否完整
- production settings 是否合规

## 实施步骤

### 1. 定义标准 span schema

新增：

```python
class RuntimeSpan(BaseModel):
    request_id: str
    trace_id: str | None
    session_key: str | None
    span_id: str
    parent_span_id: str | None
    span_type: Literal["request", "graph_node", "llm", "tool", "verification", "approval"]
    name: str
    started_at: str
    finished_at: str | None
    duration_ms: int | None
    status: Literal["ok", "error", "blocked", "fallback"]
    error_code: str | None
    attributes: dict[str, Any]
```

### 2. 标准化节点 span

在 LangGraph node enter/exit 记录：

- node name
- duration
- route decision
- output summary
- error code

### 3. 标准化 LLM span

每次 LLM 调用记录：

- scene
- model
- provider
- prompt_version
- finish_reason
- latency
- tool_call_count
- parse_status
- fallback_reason

不记录完整 prompt 和完整 LLM 原文到默认日志。

### 4. 标准化 tool span

每次工具调用记录：

- tool_name
- agent_name
- source
- operation
- risk_level
- success
- error_code
- duration
- approval_id
- contract_version

### 5. 标准化 verification span

记录：

- stage
- verifier_name
- action
- severity
- code
- patched
- blocked

### 6. 建立 metrics 聚合

先用内存或日志事件计算即可，不强依赖外部监控系统。

指标：

```text
rewrite_fallback_total
intent_unknown_total
agent_selection_clarification_total
skill_no_match_total
llm_parse_failed_total
tool_call_total
tool_failure_total
verification_block_total
approval_required_total
approval_resume_success_total
```

### 7. 建立 request replay

新增 replay service：

```python
async def build_decision_replay(request_id: str) -> DecisionReplay:
    ...
```

Replay 输出：

```text
request summary
graph path
rewrite decision
intent decision
agent decision
skill decision
tool calls
approval state
verification result
final answer
```

先基于已有存储组装：

- checkpoint snapshot
- messages
- tool_execution_logs
- approval_requests
- evidence

### 8. 建立 release readiness 脚本

新增：

```bash
uv run python scripts/check_release_readiness.py --env prod
```

检查：

- `APP_ENV=prod`
- mock tools disabled
- body fallback disabled
- LLM configured
- approval URL non-mock
- taxonomy coverage valid
- AgentCard/Skill validation passed
- prompt manifest valid
- tool contracts valid
- eval report exists or eval cases pass

### 9. 增加测试

测试覆盖：

- span schema 可构造。
- node span 包含 request_id/trace_id/node/status。
- tool span 包含 tool_name/error_code/duration。
- replay 可从 fake stores 组装。
- release readiness 能识别 mock tool。
- release readiness 能识别缺 prompt manifest。
- release readiness 能识别缺 tool contract。

## 不在本任务范围

- 不接真实 OpenTelemetry collector。
- 不做前端 trace 页面。
- 不做长期指标存储。
- 不做自动发布系统。

## 验收标准

- 关键路径能产生标准 span。
- fallback、unknown、no-skill、tool failure、verification block 可统计。
- 能通过 request_id 生成决策回放摘要。
- 有 release readiness 脚本。
- prod 环境不满足准入条件时脚本失败。
- 新增测试全部通过。
