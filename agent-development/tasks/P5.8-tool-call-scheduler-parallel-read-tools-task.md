# P5.8 Tool Call Scheduler And Parallel Read Tools Task

## 0. 任务定位

本任务优化子 Agent 内部 `ToolCallingRunner` 的工具执行性能。

当前项目中，LLM 一次返回多个 `tool_calls` 时，`ToolCallingRunner` 会按返回顺序逐个 `await ToolExecutor.execute()`。这种方式语义安全，但对多个互相独立的只读工具不够高效。

本任务目标是新增一个保守的工具调度层：

```text
LLM tool_calls
→ normalize
→ ToolCallScheduler 生成执行计划
→ ToolCallingRunner 按 plan 执行
→ ToolExecutor 仍做最终安全检查
→ observation 按原始 tool_call 顺序写回 messages
```

核心原则：

```text
ToolCallScheduler 只做调度，不做安全授权。
ToolExecutor 仍是工具能否真正执行的最终边界。
```

### 1.2 当前行为

`ToolCallingRunner.run()` 中，LLM 返回 tool calls 后：

```python
for raw_call in response.tool_calls:
    normalized = normalize_tool_call(raw_call)
    tool_result = await self.tool_executor.execute(...)
    messages.append(tool_observation)
```

当前特点：

- 一个 iteration 内多个工具串行执行；
- duplicate guard 基于 `tool_name + arguments` 统计；
- consecutive failure / same tool failure 按串行结果即时更新；
- 遇到 `human_approval_required` 立即返回，进入审批暂停；
- observation 写入顺序天然等于 LLM 原始 tool call 顺序；
- ToolExecutor 负责工具存在性、Agent 可见性、参数、权限、审批、幂等和日志。

### 1.3 当前问题

当 LLM 一次返回多个互不依赖的只读工具时，例如：

```text
query_endo_task_record(apply_seq=930...)
query_policy_update_status(policy_no=920...)
query_log_trace(request_id=req...)
```

如果这些工具都是：

```text
read
low risk
不同 resource
无需审批
无参数依赖
```

串行执行会增加一次请求内的总延迟。

## 2. 总体目标

新增：

```text
ToolCallScheduler
ToolExecutionPlan
ToolExecutionBatch
ScheduledToolCall
```

使 Runner 能够在安全条件满足时并行执行只读工具。

目标行为：

```text
只读、低风险、无审批、不同资源、无依赖的工具可以并行。
写工具、高风险、未知风险、同资源冲突、审批工具、参数非法工具默认串行或阻断。
```

第一版必须保守，不追求最大并发。

## 3. 非目标

本任务不做：

- 不改变 ToolExecutor 的安全职责；
- 不跳过 ToolExecutor；
- 不让 LLM 声明工具是否可并行；
- 不并行写工具；
- 不并行需要审批的工具；
- 不实现复杂 DAG 任务编排；
- 不让工具之间通过并行结果互相传参；
- 不改变 ToolDefinition / ToolContract 的核心含义；
- 不重写 ToolCallingRunner。

## 4. 设计原则

### 4.1 保守并行

只有满足全部条件才并行：

```text
operation == read
risk_level == low
tool_definition.is_write == false
不需要审批
参数已满足 schema
resource_key 不冲突
不是重复 tool call
UNKNOWN_TOOL_PARALLEL_POLICY 允许
```

其他情况默认串行。

### 4.2 Scheduler 不做最终安全判断

Scheduler 可以读取 ToolDefinition / ToolContract 元数据做调度判断，但不能认为工具“已经安全”。

即使 Scheduler 判定可并行，每个工具执行时仍必须进入：

```text
ToolExecutor
→ ExecutionPipeline
→ ArgumentGuard
→ Authorization
→ Verification / Precondition
→ Approval
→ Idempotency
→ Local/MCP execution
→ Tool log
```

### 4.3 observation 顺序稳定

并行执行可能乱序完成，但写回 LLM messages 时必须恢复原始 tool call 顺序：

```text
LLM tool_calls: A, B, C
实际完成顺序: B, C, A
messages 中 observation 顺序: A, B, C
```

否则下一轮 LLM 可能基于错位观察做推理。

### 4.4 审批优先中断

如果某个 batch 内任一工具返回 `human_approval_required`：

- 停止当前 runner；
- 返回审批 payload；
- 不继续执行后续 batch；
- 如果审批工具前面有已完成只读工具，保留 observation；
- 审批工具后面的工具不得提前执行。

第一版建议：任何可能审批的工具不进入 parallel batch。

## 5. 新增模型建议

新增文件：

```text
app/subagents/tool_call_scheduler.py
```

建议模型：

```python
class ScheduledToolCall(BaseModel):
    index: int
    tool_call_id: str | None
    tool_name: str
    arguments: dict[str, Any]
    operation: str | None
    risk_level: str | None
    is_write: bool
    resource_key: str | None
    can_parallel: bool
    reason: str

class ToolExecutionBatch(BaseModel):
    mode: Literal["parallel", "serial"]
    calls: list[ScheduledToolCall]
    reason: str

class ToolExecutionPlan(BaseModel):
    batches: list[ToolExecutionBatch]
    blocked_reason: str | None = None
```

可以用 dataclass 或 Pydantic，优先保持简单。

## 6. 调度规则

### 6.1 工具元数据来源

Scheduler 读取：

```text
ToolRegistry.get_definition(tool_name)
ToolDefinition.operation
ToolDefinition.risk_level
ToolDefinition.is_write
ToolDefinition.resource_type
ToolDefinition.resource_id_arg
ToolDefinition.idempotency_required
ToolDefinition.contract
ToolDefinition.metadata
```

不要从 LLM 的自然语言判断工具风险。

### 6.2 resource_key 生成

建议：

```python
resource_key = f"{resource_type}:{arguments[resource_id_arg]}"
```

如果缺少 `resource_type` 或 `resource_id_arg`：

```text
resource_key = None
```

第一版策略：

- `resource_key is None` 的 read 工具可以并行，但受配置控制；
- 写工具 `resource_key is None` 必须串行；
- 同一 `resource_key` 的工具默认串行。

### 6.3 默认并行判断表

| 条件 | 第一版策略 |
| --- | --- |
| read + low risk + no resource conflict | 可并行 |
| read + medium/high risk | 串行 |
| operation unknown | 串行 |
| risk_level unknown | 串行 |
| is_write=true | 串行 |
| idempotency_required=true | 串行 |
| 同一 resource_key | 串行 |
| 参数 normalize 失败 | 不执行，直接 observation |
| duplicate call 超阈值 | 不执行，guardrail observation |
| 可能触发审批 | 串行 |

## 7. 新增配置

修改：

```text
app/config/settings.py
.env.example
README.md
```

建议新增：

```dotenv
ENABLE_PARALLEL_READ_TOOLS=false
TOOL_PARALLEL_MAX_CONCURRENCY=4
UNKNOWN_TOOL_PARALLEL_POLICY=serial
SAME_RESOURCE_READ_PARALLEL=false
```

含义：

| 配置 | 含义 |
| --- | --- |
| `ENABLE_PARALLEL_READ_TOOLS` | 是否启用只读工具并行，默认 false |
| `TOOL_PARALLEL_MAX_CONCURRENCY` | 单个 iteration 内最大并行工具数 |
| `UNKNOWN_TOOL_PARALLEL_POLICY` | `serial` / `deny_parallel`，第一版建议只支持 `serial` |
| `SAME_RESOURCE_READ_PARALLEL` | 同一资源的 read 工具是否允许并行，默认 false |

配置治理要求：

- `TOOL_PARALLEL_MAX_CONCURRENCY >= 1`
- 未启用 `ENABLE_PARALLEL_READ_TOOLS` 时 Runner 保持当前串行行为
- prod 可以启用，但必须经过测试覆盖

## 8. ToolCallingRunner 改造方案

### 8.1 注入 Scheduler

修改：

```text
app/subagents/tool_calling_runner.py
app/bootstrap/container.py
```

构造：

```python
tool_call_scheduler = ToolCallScheduler(
    registry=tool_registry,
    enabled=settings.enable_parallel_read_tools,
    max_concurrency=settings.tool_parallel_max_concurrency,
    unknown_policy=settings.unknown_tool_parallel_policy,
    same_resource_read_parallel=settings.same_resource_read_parallel,
)
```

### 8.2 Runner 内部流程

目标伪代码：

```python
normalized_calls = normalize_all(response.tool_calls)
plan = scheduler.plan(
    agent_name=agent_name,
    agent_card=agent_card,
    normalized_calls=normalized_calls,
    existing_call_counts=tool_call_counts,
)

for batch in plan.batches:
    if batch.mode == "parallel":
        results = await execute_parallel(batch.calls)
    else:
        results = await execute_serial(batch.calls)

    append_observations_in_original_order(results)

    if any(result.needs_human_approval):
        return approval_required

    update_failure_guardrails(results)
```

### 8.3 并行执行函数

要求：

- 使用 `asyncio.Semaphore(max_concurrency)`
- 每个工具仍调用 `ToolExecutor.execute`
- 单个工具异常要转换成 ToolResult 风格 observation，不能让 gather 整体失败
- 使用 `asyncio.gather(..., return_exceptions=True)` 或内部捕获异常
- 执行完成后按 `ScheduledToolCall.index` 排序

## 9. 失败保护重构

当前失败保护是串行即时更新。并行后需要按 batch 结果统一更新。

必须保留：

```text
max_consecutive_tool_failures
max_same_tool_failures
max_duplicate_tool_calls
```

建议：

- duplicate guard 在执行前检查；
- same tool failure 在结果回收后按 tool_call_key 更新；
- consecutive failure 按 observation 写回顺序计算；
- 任一 guardrail 命中后停止后续 batch。

## 10. 审批兼容

审批工具必须满足：

- 默认不进入 parallel batch；
- 一旦返回 `human_approval_required`，停止后续工具；
- pending_tool_call 保留原 tool_call_id/name/arguments；
- ApprovalStore resume_state 仍能恢复；
- 审批 callback 后继续原有恢复链。

必须回归：


## 14. 验收标准

必须满足：

1. 默认配置下行为与当前串行执行一致。
2. 开启 `ENABLE_PARALLEL_READ_TOOLS=true` 后，独立 read 工具可并行。

4. 同一 resource_key 默认不并行。
5. 需要审批的工具不并行，审批链不被绕过。
6. ToolExecutor 仍是唯一执行入口。
7. observation 按 LLM 原始 tool_call 顺序写入。
8. duplicate / failure guardrail 仍生效。
9. `uv run pytest -q` 通过。
10. `uv run python scripts/run_agent_evals.py` 通过。

## 15. 风险

| 风险 | 控制 |
| --- | --- |
| 并行导致写操作竞态 | 写工具默认串行 |
| LLM 读到乱序 observation | 按原始 index 排序写回 |
| 审批被绕过 | 审批工具不并行，ToolExecutor 保持最终判断 |
| guardrail 统计错误 | batch 结果按 observation 顺序更新 |
| resource_key 缺失误并行 | 第一版 unknown/写工具串行 |
| 测试只覆盖 happy path | 增加失败、审批、同资源、异常 case |

## 16. 需要人工确认的问题

1. 第一版是否默认关闭 `ENABLE_PARALLEL_READ_TOOLS`。
2. 同一资源的只读工具是否永远串行，还是后续允许配置开启。
3. `operation/risk_level` 缺失时是否只串行，还是直接拒绝并行。
4. 是否需要在 ToolContract 中补充更明确的 `parallel_safe` 字段。
