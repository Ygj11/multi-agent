# P5.5 AgentCard Interface And Nacos Registry Task

优先级：企业级 P5.5

## 任务定位

本任务用于将当前项目内部使用的 `AgentCard`，逐步扩展为可对外暴露、可注册发现、可被其他系统识别的 Agent 能力描述。

当前项目已经有 `app/agents/cards/*.yaml` 和 `app/schemas/agent_card.py`，但它们的定位是 **内部路由和执行治理配置**：

- MainGraph 根据 `supported_routes` 选择子 Agent。
- AgentSelection 使用 `capabilities`、`examples`、`required_entities` 做候选打分。
- ToolRegistry / ToolExecutor 使用 `private_tools`、`public_tools_allowed`、`mcp_policy.enabled` 做工具可见性控制。
- SkillCatalog 使用 `skills` 关联可加载的业务 SOP。
- Authorization 使用 `access_policy` 控制 Agent 入口权限。

`agent-development/img/interface/agentCard*.jpg` 中的 AgentCard 定义更偏向 **对外智能体服务描述卡**：

- 描述一个智能体服务叫什么。
- 服务地址是什么。
- 版本是什么。
- 支持哪些输入输出模式。
- 支持哪些能力协议。
- 如何认证。
- 暴露哪些技能摘要。
- 由哪个 provider 提供。

因此，本任务不应直接把当前内部 AgentCard 改成图片里的结构，而应建立一层 **Public AgentCard 适配与发布机制**。

## 当前结论

当前项目 **语义上部分具备 AgentCard 能力**，但 **接口格式不符合图片定义**。

### 已具备

- Agent 名称：`agent_name` / `display_name`
- Agent 描述：`description`
- 版本：`version`
- 能力描述：`capabilities`
- 支持路由：`supported_routes`
- Skill 绑定：`skills`
- 示例：`examples`
- 权限边界：`access_policy`
- 私有工具边界：`private_tools`
- MCP 工具入口边界：`mcp_policy.enabled`
- RAG namespace：`rag_namespaces`
- Memory policy：`memory_policy`

### 不符合图片定义

| 图片字段 | 当前项目字段 | 问题 |
| --- | --- | --- |
| `name` | `agent_name` / `display_name` | 字段名和语义未统一 |
| `url` | 无 | 缺少对外服务 URL |
| `documentationUrl` | 无 | 缺少文档链接 |
| `provider` | 无 | 缺少 provider 描述 |
| `capabilities: Map` | `capabilities: list[str]` | 类型和语义不一致 |
| `authentication` | `access_policy` | 一个是对外认证描述，一个是内部权限策略 |
| `defaultInputModes` | 无 | 缺少输入模式 |
| `defaultOutputModes` | 无 | 缺少输出模式 |
| `skills: List<Skill>` | `skills: list[str]` | 当前只是 skill_id，不是对外 Skill 对象 |
| Skill `id/name/description/tags/examples/inputModes/outputModes` | 分散在 `SKILL.md` frontmatter | 需要通过 SkillCatalog 展开 |

## 核心设计原则

### 1. 内外分离

内部 AgentCard 不直接等同于对外 AgentCard。

```text
Internal AgentCard
  负责内部路由、权限、工具、Skill 绑定、RAG 和 memory

Public AgentCard
  负责对外服务发现、能力描述、输入输出协议和认证摘要
```

### 2. 内部主流程不被外部协议污染

不为了符合图片字段，强行修改当前主流程字段。

禁止把下面这些内部治理字段直接暴露给外部：

- `private_tools`
- MCP 工具风险治理细节
- `access_policy` 细节
- `memory_policy` 细节
- 完整 `SKILL.md` 内容
- 内部 prompt
- 内部 routing policy
- 工具真实参数细节

### 3. Public AgentCard 由内部配置投影生成

对外 AgentCard 应通过 projector 生成，而不是手写两套完全独立配置。

```text
Internal AgentCard YAML
    +
SkillCatalog metadata
    +
Public interface config
    ↓
Public AgentCard
```

### 4. 先本地接口，后注册中心

先实现本地 `/api/agents/cards` 能正确返回图片定义格式，再接入 Nacos。

否则容易把“AgentCard 对外格式”和“Nacos 注册发现”两个问题混在一起。

### 5. Nacos 只做注册发现，不参与运行时决策

Nacos 不能替代：

- MainGraph 路由
- Intent Recognition
- AgentSelection
- SkillSelection
- ToolExecutor 权限校验
- Approval
- Verification

Nacos 只用于让外部系统发现当前服务和 Agent 能力。

## Nacos 思想下的定位

当前项目中的子 Agent 不是独立微服务，而是同一个 FastAPI 进程中的内部执行模块。

因此，不建议把每个子 Agent 都注册成一个独立 Nacos 服务实例。

推荐模型：

```text
Nacos Service:
  health-agent-platform

Service Instance Metadata:
  agent_names: pos_query_agent,troubleshooting_agent
  version: <platform_version>
  protocol: http
  agent_card_index: agent-cards.index.json

Nacos Config:
  dataId: agent-card.pos_query_agent.json
  dataId: agent-card.troubleshooting_agent.json
  dataId: agent-cards.index.json
```

也就是说：

- 服务发现层面注册的是 Agent 平台服务实例。
- 配置中心层面发布的是每个 Agent 的 Public AgentCard。
- 外部系统先发现平台服务，再读取 AgentCard 判断能力。

## 阶段二：新增 Public AgentCard Schema

建议新增：

```text
app/schemas/public_agent_card.py
```

建议 schema：

```python
class PublicAgentProvider(BaseModel):
    organization: str | None = None
    url: str | None = None


class PublicAgentCapabilities(BaseModel):
    streaming: bool = False
    pushNotifications: bool = False
    stateTransitionHistory: bool = False


class PublicAgentAuthentication(BaseModel):
    schemes: list[str] = Field(default_factory=list)
    credentials: str | None = None


class PublicAgentSkill(BaseModel):
    id: str
    name: str
    description: str
    tags: list[str] = Field(default_factory=list)
    examples: list[str] = Field(default_factory=list)
    inputModes: list[str] = Field(default_factory=lambda: ["text"])
    outputModes: list[str] = Field(default_factory=lambda: ["text"])


class PublicAgentCard(BaseModel):
    name: str
    description: str
    url: str
    version: str
    documentationUrl: str | None = None
    provider: PublicAgentProvider | None = None
    capabilities: PublicAgentCapabilities = Field(default_factory=PublicAgentCapabilities)
    authentication: PublicAgentAuthentication | None = None
    defaultInputModes: list[str] = Field(default_factory=lambda: ["text"])
    defaultOutputModes: list[str] = Field(default_factory=lambda: ["text"])
    skills: list[PublicAgentSkill]
```

字段名应优先贴近图片定义。

如需内部 Python 风格字段，可使用 Pydantic alias，但对外 JSON 必须输出图片定义字段。

## 阶段三：扩充 AgentCard YAML

在现有 `app/agents/cards/*.yaml` 中新增一个独立块。

示例：

```yaml
interface:
  name: 保全实时查询智能体
  url: /api/chat
  documentationUrl: /docs
  provider:
    organization: PingAn Health
    url: ""
  capabilities:
    streaming: false
    pushNotifications: false
    stateTransitionHistory: true
  authentication:
    schemes:
      - bearer
    credentials: null
  defaultInputModes:
    - text
  defaultOutputModes:
    - text
```

### 字段规则

- `interface.name` 默认可取 `display_name`。
- `interface.url` 默认可取统一平台入口 `/api/chat`。
- `interface.version` 默认取内部 `version`。
- `interface.description` 默认取内部 `description`。
- `interface.defaultInputModes` 默认 `["text"]`。
- `interface.defaultOutputModes` 默认 `["text"]`。
- `interface.capabilities` 是协议能力，不是业务能力标签。
- 内部 `capabilities: list[str]` 仍然保留，继续作为路由/语义判断证据。

### 发布规则

对于每个 Public AgentCard：

```text
dataId = agent-card.<agent_name>.json
group = AGENT_PLATFORM
content = PublicAgentCard JSON
```

同时发布索引：

```text
dataId = agent-cards.index.json
content = {
  "agents": [
    {
      "name": "pos_query_agent",
      "dataId": "agent-card.pos_query_agent.json",
      "version": "1.0.0"
    }
  ]
}
```

服务发现：

```text
serviceName = health-agent-platform
metadata = {
  "agent_names": "pos_query_agent,troubleshooting_agent",
  "agent_card_index": "agent-cards.index.json",
  "version": "<platform_version>"
}
```

### 启动行为

项目启动时：

```text
加载 Internal AgentCard
校验 taxonomy
校验 SkillCatalog
生成 Public AgentCard
如果 ENABLE_AGENT_CARD_REGISTRY=true：
    发布到 registry
否则：
    只保留本地 API 可查询
```

### 失败策略

默认：

- Nacos 发布失败不影响本地服务启动。
- 打印结构化 warning 日志。
- API 仍可返回本地 Public AgentCard。

生产可配置：

```env
AGENT_CARD_REGISTRY_FAIL_FAST=false
```

如果为 true：

- Nacos 发布失败时启动失败。

## 风险点

### 1. 字段语义混乱

最容易出错的是 `capabilities`。

内部：

```yaml
capabilities:
  - task_status_diagnosis
  - internal_log_analysis
```

对外：

```yaml
capabilities:
  streaming: false
  pushNotifications: false
  stateTransitionHistory: true
```

二者不是同一个东西。

必须通过 schema 命名和 projector 明确区分。

### 2. Skill 暴露过多

对外只暴露 Skill 摘要，不暴露完整 SOP。

原因：

- `SKILL.md` 中可能包含内部处理规则。
- 可能包含工具调用策略。
- 可能包含不应给外部系统的排障细节。

### 3. Nacos 变成强依赖

默认必须关闭。

本地开发、单元测试和 CI 不应依赖 Nacos。

### 4. 外部 URL 配置不准确

`interface.url` 如果写死 `/api/chat`，外部系统可能不知道 host。

建议 projector 支持：

```env
PUBLIC_AGENT_BASE_URL=https://agent.example.com
```

最终输出：

```text
https://agent.example.com/api/chat
```

### 5. 每个子 Agent 是否独立服务

当前不是。

不要在 Nacos 中注册：

```text
pos_query_agent
troubleshooting_agent
```

作为独立服务实例。

推荐只注册：

```text
health-agent-platform
```

然后通过 metadata / config 暴露其内部 Agent 能力。
