# P5.10 Prompt Output Contract Injection Task

## 任务定位

当前项目已经通过 `app/llm/output_schemas.py` 对多个 LLM 结构化输出做了 Pydantic 强校验，例如：

- `QueryRewriteLLMOutput`
- `IntentRecognitionLLMOutput`
- `AgentSelectionLLMOutput`
- `SkillSelectionLLMOutput`
- `TaskCompletionLLMOutput`

但 prompt 侧大多只写了：

```text
Return strict JSON only with these keys...
```

或：

```text
字段符合 TaskCompletionLLMOutput
```

这意味着：

```text
代码是强校验
Prompt 是软约束
```

真实 LLM 并不知道 Python/Pydantic 类的字段、类型、枚举、嵌套结构和一致性规则。模型只能靠自然语言猜输出格式，容易出现：

- 少字段；
- 多字段；
- 字段类型错误；
- 枚举值错误；
- 嵌套 `repair_plan` 结构错误；
- `completed=true` 但 `status=CONTINUE`；
- Markdown 包裹 JSON；
- 输出解释文本或思维链；
- 触发格式修复，甚至进入 `HUMAN_HANDOFF`。

本任务目标是建立统一的 **Prompt Output Contract 注入机制**，让 prompt 看到和代码强校验一致的输出契约。

---

## 总体目标

建立如下链路：

```text
PromptManifest.output_schema
        ↓
SCHEMA_REGISTRY
        ↓
Pydantic JSON Schema / 精简示例 JSON / 枚举说明
        ↓
PromptLoader 自动注入 {output_contract}
        ↓
LLM 按明确契约输出
        ↓
parse_llm_json_schema 强校验
```

完成后，结构化 LLM scene 的 prompt 不再只写 schema 名称，而是明确包含：

1. 允许输出的字段；
2. 字段类型；
3. 必填/可选；
4. 枚举合法值；
5. 嵌套对象结构；
6. 关键一致性规则；
7. 一个可直接仿照的 JSON 示例；
8. 明确禁止 Markdown、解释文本和额外字段。

## 设计原则

### 1. Schema 是唯一事实来源

输出契约必须从 `SCHEMA_REGISTRY` 中的 Pydantic 模型生成。

禁止在每个 prompt 中手写一份独立 JSON Schema，避免：

```text
Pydantic schema 改了
Prompt 里的格式说明没改
```

### 2. Contract 给 LLM 看，Pydantic 仍做最终裁判

`output_contract` 是 prompt 指导，不是安全边界。

最终仍必须由：

```python
parse_llm_json_schema(response.content, TargetSchema)
```

完成强校验。

### 3. 不把完整 Pydantic JSON Schema 原样塞进 prompt

Pydantic 的 `model_json_schema()` 会比较长，也可能包含 `$defs`、`anyOf` 等模型不易遵循的结构。

本任务应生成 **LLM 友好的精简契约**：

```text
Output schema name
Allowed JSON object shape
Field list
Enum values
Nested object shape
Example JSON
Important validation rules
No extra keys
No Markdown
```

### 4. Prompt 模板只引用 `{output_contract}`

每个结构化 prompt 的 system template 中统一写：

```text
{output_contract}
```

不要在每个 prompt 中重复维护完整字段结构。

### 5. 渐进接入，不一次性改所有非结构化 scene

第一阶段只接入：

- `query_rewrite`
- `intent_recognition`
- `agent_selection`
- `skill_selection`
- `task_completion_verifier`

暂不接入：

- `memory_summary`：文本输出；
- `subagent_reasoning`：tool calling + natural language answer；
- `final_compliance`：需要单独审查现有 verifier schema 和 prompt。

## 目标架构

新增：

```text
app/prompts/output_contract.py
```

核心职责：

```text
1. 根据 PromptSceneManifest.output_schema 找到 Pydantic 模型；
2. 生成面向 LLM 的输出契约文本；
3. 生成示例 JSON；
4. 展开关键枚举合法值；
5. 展开嵌套 BaseModel，例如 RepairPlan；
6. 对 text/SubAgentResult/VerificationResult 等非本任务 schema 返回空或简短说明；
7. 供 PromptLoader 自动注入。
```

推荐类：

```python
class PromptOutputContractRenderer:
    def render_for_scene(self, scene: PromptSceneManifest) -> str:
        ...

    def render_for_schema(self, schema_name: str) -> str:
        ...
```

`PromptLoader.render_scene_system()` / `render_scene_user()` 应自动补充变量：

```python
variables.setdefault(
    "output_contract",
    self.output_contract_renderer.render_for_scene(self.scene(scene)),
)
```

这样调用方不需要每次手动传 `output_contract`。

## 输出契约格式建议

以 `TaskCompletionLLMOutput` 为例，渲染结果应接近：

```text
Output contract: TaskCompletionLLMOutput

Return exactly one JSON object. Do not wrap it in Markdown. Do not add extra keys.

Allowed fields:
- status: string enum, required. One of: PASS, CONTINUE, NEED_USER, HUMAN_HANDOFF, FAILED.
- completed: boolean, required.
- summary: string, required.
- completed_items: array of string, optional, default [].
- missing_items: array of string, optional, default [].
- repair_plan: object or null, optional.
- confidence: number, required, 0.0 <= confidence <= 1.0.
- reasoning_summary: string, optional. Short auditable reason only. Do not include hidden chain-of-thought.
- evidence_ids: array of string, optional, default [].

repair_plan when status is CONTINUE:
- reason: string, required.
- completed_items: array of string.
- missing_items: array of string.
- next_steps: array of string.
- do_not_repeat: array of string.
- reuse_evidence_ids: array of string.
- expected_new_evidence: array of string.
- target_agent: string, required. Must equal selected_agent from input.
- selected_skill_id: string, required. Must equal selected_skill_id from input.
- confidence: number, required, 0.0 <= confidence <= 1.0.
- fingerprint: string or null.

Validation rules:
- If status is PASS, completed must be true and repair_plan must be null.
- If status is CONTINUE, completed must be false and repair_plan is required.
- If status is NEED_USER, HUMAN_HANDOFF, or FAILED, completed must be false.
- Do not output fields outside this contract.

Example:
{
  "status": "PASS",
  "completed": true,
  "summary": "已基于工具证据确认任务完成。",
  "completed_items": ["已查询任务状态"],
  "missing_items": [],
  "repair_plan": null,
  "confidence": 0.86,
  "reasoning_summary": "工具结果和证据摘要均支持任务已完成。",
  "evidence_ids": []
}
```

## Phase 1：新增 PromptOutputContractRenderer

### 目标

## 风险点

### 1. Prompt 变长

注入 contract 会增加 prompt token。

控制方式：

- 输出精简 contract；
- 不把完整 `$defs` 原样塞入；
- 不给文本和 tool calling scene 注入大段 contract。

### 2. Contract 与业务规则重复

schema contract 只讲输出结构，业务边界仍留在 prompt 正文，例如：

- intent 不选择 agent；
- query rewrite 不选择工具；
- verifier 不执行工具；
- repair 不换 agent/skill。

### 3. 嵌套模型过度展开

第一阶段只需要支持当前结构化 schema 中用到的嵌套模型，尤其是 `RepairPlan`。

不要为了未来所有 Pydantic 类型写复杂 schema compiler。

### 4. 测试假阳性

不能只测试 `output_contract` 字符串存在，还要测试关键字段、枚举和示例 JSON。

---

## 验收标准

完成后必须满足：

1. `PromptOutputContractRenderer` 可以从 `SCHEMA_REGISTRY` 生成稳定 contract；
2. `PromptLoader.render_scene_system()` 自动注入 `{output_contract}`；
3. `task_completion_verifier` prompt 明确展示 `TaskCompletionLLMOutput` 结构和 `RepairPlan` 结构；
4. `query_rewrite` prompt 明确展示 `QueryRewriteLLMOutput` 字段和 `rewrite_type` 枚举；
5. `intent_recognition` prompt 明确展示 `IntentRecognitionLLMOutput` 字段，且不输出 entities；
6. `agent_selection` / `skill_selection` prompt 明确展示输出字段；
7. Prompt Eval 静态校验继续通过；
8. Prompt Eval provider 模式使用注入后的 contract；
9. Agent Dynamic E2E Eval 使用注入后的 contract；
10. 所有现有测试通过；
11. 不改变业务路由和 API 响应。
