# P4.1 Enterprise Fallback And No-Skill Semantics Task

优先级：企业级 P0

## 任务定位

本任务处理两个企业级 Agent 项目中非常关键的问题：

1. silent fallback：主路径失败后系统静默走兜底，开发者和监控无法知道真实原因。
2. no-skill semantics：Skill 未命中时系统是否还允许子 Agent 泛化执行。

这两个问题都容易造成“看起来成功，实际质量不可控”的风险。

## 当前问题

### 1. LLM 失败后会静默 fallback

当前多个节点存在类似行为：

- Query Rewrite LLM JSON 解析失败，直接返回 `None`，然后走规则 rewrite。
- Intent Recognition LLM JSON 解析失败或 intent 非法，直接返回 `None`，然后走规则 intent。
- Agent Selection LLM router 不可用，fallback 到 rule top1。
- Skill rerank 失败，fallback 到 rule top1 或 no skill。

这些 fallback 本身不一定错，但不能 silent。

### 2. Skill 未命中时仍可能 generic execution

`SkillContextResolver` 中存在：

```text
generic_skill_content = "No specific Skill matched confidently..."
```

当 `selected_skill_id is None` 时，当前代码会继续构建 subagent context，让 agent 依靠 AgentCard、上下文和可见工具继续执行。

这不是 `is_default=true` 的默认 skill，但行为上仍是一种泛化兜底执行。

用户之前已经明确过：不需要默认 skill，skill 匹配到就是匹配成功，匹配不到就不需要 skill，不要 skill 的兜底方案。

### 3. SubAgent prompt 太薄

当前 `subagent_reasoning/system.md` 只做了很基础的说明：

- use only provided tools
- enough evidence available then answer
- follow selected skill body when present

企业级工具调用需要更强约束。

### 4. BaseSubAgent 中存在业务硬编码

`BaseSubAgent._diagnosis_from_runner`、`_recommendation_from_runner`、`_responsibility_from_runner` 中包含 troubleshooting/E102 类语义。

通用基类不应持有具体业务诊断话术。

## 架构目标

### 1. 所有 fallback 必须可观测

每个 fallback 至少记录：

```text
fallback_used: true
fallback_source: query_rewrite | intent_recognition | agent_selection | skill_selection | llm_provider
fallback_reason: llm_disabled | llm_error | llm_json_parse_failed | invalid_intent | invalid_sub_intent | low_confidence | no_candidate
llm_status: disabled | success | provider_error | parse_failed | invalid_output
```

### 2. no-skill 行为必须显式配置

新增：

```text
NO_SKILL_POLICY=clarify|answer_no_skill|generic_dev_only
```

建议默认：

```text
local: generic_dev_only 或 clarify
test: clarify
staging: clarify
prod: clarify
```

策略含义：

- `clarify`：无 skill 命中时返回澄清，不进入 tool loop。
- `answer_no_skill`：返回“当前 agent 没有匹配到可处理技能”的安全回答。
- `generic_dev_only`：仅本地开发可用，继续 generic execution。

### 3. 子 Agent prompt 具备工具调用约束

Prompt 必须约束：

- 不得编造工具结果。
- 工具结果不足时要说明缺少哪些证据。
- 写工具必须等待系统审批。
- 不得原样输出完整 JSON。
- 工具失败时按错误类型说明。
- 最终回答要包含诊断、证据、下一步动作和不确定性。

### 4. 通用基类不包含具体业务诊断话术

`BaseSubAgent` 只做模板逻辑，不写具体业务结论。

## 实施步骤

### 1. 定义统一 fallback/LLM 状态枚举

新增稳定错误码：

```text
llm_disabled
llm_provider_error
llm_json_parse_failed
llm_invalid_output
taxonomy_mismatch
invalid_intent
invalid_sub_intent
agent_router_unusable
skill_rerank_unusable
no_confident_skill
no_skill_policy_blocked
```

### 2. 扩展结果模型

为关键结果增加可选字段：

- `QueryRewriteResult`
- `IntentResult`
- `AgentSelectionResult`
- `SkillSelectionResult`

建议字段：

```python
llm_status: str | None
fallback_used: bool = False
fallback_reason: str | None = None
decision_trace: dict[str, Any] = {}
```

如果不想一次性改所有 schema，也可以先通过 metadata/log_event 记录，但长期建议 schema 化。

### 3. Query Rewrite fallback 可观测

当前 `_rewrite_with_llm()` 返回 `None` 的场景要区分：

- provider disabled
- provider error
- empty content
- JSON parse failed

规则 fallback 结果必须带：

```text
fallback_used=true
fallback_reason=...
llm_status=...
```

### 4. Intent Recognition fallback 可观测

当前 `_recognize_with_llm()` 返回 `None` 的场景要区分：

- LLM disabled
- JSON parse failed
- intent not allowed
- sub_intent invalid
- confidence too low

规则 fallback 结果必须带 fallback reason。

### 5. Agent Selection fallback 可观测

LLM router 不可用时：

- 不只写 reason_suffix。
- 记录 `llm_status` 和 `fallback_reason=agent_router_unusable`。

### 6. Skill Selection fallback 可观测

LLM rerank 不可用、低置信、非法 skill_id、无 skill 命中都要区分原因。

### 7. 增加 `NO_SKILL_POLICY`

Settings 新增：

```python
no_skill_policy: Literal["clarify", "answer_no_skill", "generic_dev_only"] = "clarify"
```

`SkillContextResolver.resolve()` 中：

- `clarify`：返回 `need_clarification=True`，不加载 `generic_skill_content`。
- `answer_no_skill`：返回 no-skill 安全 answer 所需 metadata。
- `generic_dev_only`：只有 `APP_ENV=local` 才允许继续 generic execution。

### 8. 修改 BaseSubAgent 对 no-skill 的处理

当 sub_context 表示 no-skill blocked 时：

- 不进入 tool loop。
- 返回 clarification 或安全回答。

### 9. 强化 subagent prompt

更新 `app/prompts/subagent_reasoning/system.md`，增加：

- 工具调用原则
- 证据原则
- 写工具审批原则
- 工具失败处理
- 答案结构
- 禁止事项

### 10. 移除 BaseSubAgent 业务硬编码

从 `BaseSubAgent` 移除或弱化：

- `_diagnosis_from_runner`
- `_recommendation_from_runner`
- `_responsibility_from_runner`

如果仍需要默认字段，应返回 `None`，具体业务放到 skill 或具体 subagent。

### 11. 增加测试

测试覆盖：

- Query Rewrite LLM invalid JSON -> rule fallback 且记录 reason。
- Intent LLM invalid intent -> fallback 且记录 reason。
- Agent router invalid output -> fallback 且记录 reason。
- Skill rerank invalid skill_id -> fallback 且记录 reason。
- no skill + `NO_SKILL_POLICY=clarify` -> 不进入 tool loop。
- no skill + `NO_SKILL_POLICY=generic_dev_only` + `APP_ENV=prod` -> 启动或执行失败。
- SubAgent prompt render 包含工具和证据约束关键语句。

## 不在本任务范围

- 不做 PromptManifest。
- 不做 eval runner。
- 不做 ToolContract。
- 不外置 query/intent 规则。
- 不做全链路 observability dashboard。

## 验收标准

- 不再存在无痕 LLM fallback。
- no skill 命中时不会默认 generic execution。
- 生产环境不能启用 `generic_dev_only`。
- subagent prompt 对工具调用、证据和失败处理有明确约束。
- 通用 BaseSubAgent 不再包含具体 E102/troubleshooting 业务结论。
- 新增测试全部通过。
