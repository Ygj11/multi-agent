# P5.6 Runtime Application Container Refactor Task

## 任务定位

当前项目的运行时装配主要集中在 `app/main.py:create_app()` 中。FastAPI 入口同时承担了配置读取、存储初始化、LLM 初始化、ToolRegistry 注册、Skill/Agent 校验、Graph 构建、Orchestrator 构建、MCP startup 等职责。

这在项目早期可以接受，但随着系统已经包含 FastAPI、LangGraph、SubAgent、ToolExecutor、Approval、Verification、MCP、Memory、Checkpoint 等模块，长期应把“Agent Runtime 是应用本体，FastAPI 只是入口之一”的边界拉清楚。

本任务的目标是新增统一的 Runtime Application Container：

```text
build_app_container(settings)  # 同步：确定性对象装配与本地静态初始化
await container.startup()      # 异步：外部资源启动、发现、校验
await container.shutdown()     # 异步：真实可用资源释放边界
```

## 背景问题

当前 `app/main.py:create_app()` 中混合了三类职责：

1. 同步确定性装配与本地静态初始化：
   - `Settings`
   - storage/stores
   - LLM provider
   - knowledge service
   - ToolRegistry
   - ToolExecutor
   - SkillCatalog
   - AgentCardLoader
   - Graph
   - Orchestrator

2. 运行时启动动作：
   - MCP client initialize
   - MCP tools 注册到 ToolRegistry
   - tool contract 校验并记录日志

3. FastAPI 适配：
   - `FastAPI(...)`
   - `app.state.*`
   - route 定义
   - FastAPI dependency 注入
   - HTTP request/response adapter

这些职责混在一起会导致：

- 后续 CLI 如果直接跑 Orchestrator，需要复制一份初始化。
- 测试如果想跑真实 runtime，也容易和 FastAPI 强绑定。
- FastAPI 入口越来越大，不容易看出系统真正的 runtime 依赖关系。
- MCP、Tool、Approval、Verification 等启动逻辑分散在入口里，边界不够清楚。
- 新增入口或后台任务时，可能出现 FastAPI 和非 FastAPI 行为漂移。

## 目标架构

新增一个容器模块，建议位置：

```text
app/bootstrap/container.py
```

核心对象建议：

```python
@dataclass(slots=True)
class AppContainer:
    settings: Settings
    storage: StorageBundle
    llm_provider: LLMProvider | None
    knowledge_service: KnowledgeService
    tool_registry: ToolRegistry
    tool_executor: ToolExecutor
    tool_calling_runner: ToolCallingRunner
    verification_service: VerificationService
    approval_service: ApprovalService
    session_manager: SessionManager
    skill_catalog: SkillCatalog
    context_builder: ContextBuilder
    subagent_manager: SubAgentManager
    agent_card_loader: AgentCardLoader
    graph: CompiledStateGraph
    orchestrator: AgentOrchestrator
    _started: bool = False
    async def startup(self) -> None: ...
    async def shutdown(self) -> None: ...
```

实际字段以当前真实类型为准，不为了形式引入过多泛型或抽象。

`RequestAdapter` / `ResponseAdapter` 当前直接依赖 `ChatRequest` / `ChatResponse` 等 API schema，更接近入口适配层，不是 Agent Runtime 核心领域组件。本任务默认不把它们作为 RuntimeContainer 核心字段；如果实现时为了低风险临时保留，必须在代码和报告中标注它们是入口适配器，而不是 Runtime 内部依赖。

## 设计原则

### 1. 同步 build

`build_app_container()` 做同步确定性装配与本地静态初始化。它不是数学意义上的无副作用纯函数，因为当前 `build_storage()` 会创建 SQLite 路径并执行本地建表 SQL。

允许在 `build_app_container()` 中执行：

- 读取或接收 `Settings`
- 加载本地 YAML、Markdown、AgentCard、Skill metadata
- 初始化本地 SQLite 和 Store
- 构建 store/service/manager/registry/graph/orchestrator
- 注册 local public/private/admin tools
- 执行不需要 await 的静态校验

禁止在 `build_app_container()` 中执行：

- MCP 网络发现
- 外部 API health check
- LLM 调用
- 远程配置拉取
- 需要 `await` 的资源启动
- 长耗时外部资源启动

### 2. 异步 startup

`container.startup()` 做应用进入可服务状态前的异步启动：

- MCP client initialize
- MCP capability registry 刷新
- MCP tools 注册到 ToolRegistry
- MCP 注册后的 tool contract 校验
- 必要的启动日志

如果未来有数据库连接池、远程配置、外部健康检查，也应放入 startup。

必须定义清晰的幂等语义：

- `AppContainer` 维护 `_started: bool = False`。
- 第一次 startup 执行 MCP 初始化、能力发现、工具注册和动态 contract 校验。
- 重复 startup 直接返回，不重复初始化和注册。
- startup 失败时不得把 `_started` 标记为 `True`。
- 必须在所有必要启动步骤完成后，才设置 `_started=True`。

### 3. 异步 shutdown

`container.shutdown()` 负责释放异步资源：

- 当前 `MCPClientManager` 尚无统一 close/shutdown 协议，第一阶段 `shutdown()` 可以是结构性 no-op 或有限状态清理。
- 未来数据库连接池、HTTP client session、后台任务都放在这里关闭。
- 未 startup 调用 shutdown 必须安全 no-op。
- startup 成功后 shutdown 应恢复生命周期状态。

不得在文档或交付报告中声称“已关闭所有 MCP 长连接并释放所有远程资源”，除非真实实现了对应 close/shutdown。

### 4. AppContainer 只能存在于系统最外层

AppContainer 可以被以下位置使用：

- FastAPI `create_app()`；
- FastAPI lifespan；
- CLI 入口；
- Worker 入口；
- 集成测试；
- 运维健康检查。

内部业务组件不得接收或依赖整个 AppContainer。

禁止出现：

```python
class ToolCallingRunner:
    def __init__(self, container: AppContainer):
        self.tool_executor = container.tool_executor
        self.llm_provider = container.llm_provider
```

内部组件必须继续使用显式依赖：

```python
tool_calling_runner = ToolCallingRunner(
    llm_provider=llm_provider,
    tool_executor=tool_executor,
)
```

否则 Container 会退化为 Service Locator 或“上帝对象”，导致组件真实依赖不清晰。

### 5. FastAPI 只做 HTTP 入口

`app/main.py:create_app()` 应变成：

```text
settings = get_settings()
container = build_app_container(settings, sqlite_db_path=...)

FastAPI lifespan:
    await container.startup()
    try:
        yield
    finally:
        await container.shutdown()

FastAPI app 构造阶段:
    app.state.container = container
    不再平铺挂载 app.state.xxx

routes:
    从闭包或 app.state.container 获取 orchestrator、approval_service 等 runtime 能力
    RequestAdapter / ResponseAdapter 保留在 API adapter 层
```

FastAPI 不再直接负责 ToolRegistry、Graph、Approval、Verification 的装配细节。

## startup 迁移清单

从当前 FastAPI lifespan 迁移到 `AppContainer.startup()`：

```python
if settings.enable_mcp_client:
    await mcp_client_manager.initialize()
    tool_registry.register_mcp_tools(mcp_capability_registry.list_tools())
    contract_errors = tool_registry.validate_contracts(strict=settings.app_env == "prod")
    log_event(...)
```

要求：

- startup 可重复调用时不得重复初始化 MCP client，也不得重复注册 MCP tools。
- 如果当前 `ToolRegistry.register_mcp_tools()` 是覆盖式注册，可以在任务中说明其幂等边界；但 Container 仍应通过 `_started` 避免重复 startup。
- MCP 初始化失败策略保持当前行为，不在本任务改变。
- contract warning / prod strict 行为保持不变。
- 当前 MCP 初始化发生在 FastAPI lifespan `yield` 之前，因此会阻塞应用完成 startup；但单个 MCP Server 失败时，当前策略是记录 warning 并继续启动。
- 不要把本任务顺手改成后台异步发现 MCP 工具，因为这会改变工具可见性和启动一致性语义。

## app.state container-only 策略

为了避免 `app.state.xxx` 形成新的历史包袱，最终只允许 `create_app()` 挂载 `app.state.container`。

挂载时机必须在 `create_app()` 构造阶段完成，而不是等到 lifespan startup 后才挂载。这样可以保证：

```text
app = create_app()
assert app.state.container is not None
```

不依赖 TestClient 是否触发 lifespan。

例如：

```python
app.state.container = container
```

验收时需确认：

- 代码和测试不再直接读取 `app.state.approval_store`、`app.state.tool_registry` 等平铺字段。
- 需要 runtime 组件时统一通过 `app.state.container` 访问。
- `app.state._state` 中除 `container` 外不应存在 runtime 组件平铺字段。

## 非目标

本任务不做以下事情：

- 不实现 CLI。
- 不修改 `/api/chat` 请求体或响应体。
- 不改变 MainGraph 节点顺序。
- 不改变 Query Rewrite、Intent、Agent Selection、Skill Selection 逻辑。
- 不重构 ToolExecutor / Approval / Verification 内部行为。
- 不新增 DI 框架。
- 不引入复杂 IoC container。
- 不引入复杂 Bundle 分层，例如 ToolBundle/AgentBundle/GraphBundle。
- 不让 Graph Node、SubAgent、ToolExecutor、Service 或 Store 依赖 AppContainer。
- 不把所有模块改成 class provider。
- 不改变 MCP 工具暴露策略。
- 不实现 MCP 后台异步刷新。
- 不新增 MCP close/shutdown 协议，除非当前真实 client 已支持。
- 不改变测试数据和业务 AgentCard/Skill 配置。

## 推荐实施步骤

### Step 1：新增 AppContainer schema

新增：

```text
app/bootstrap/container.py
```

定义：

- `AppContainer`
- `build_app_container(...)`

`AppContainer` 优先用 dataclass，不引入 Pydantic，避免 callable、compiled graph、store 等对象被模型序列化约束。

`AppContainer` 字段可以覆盖当前顶层 runtime 组件，但不要为了形式拆分复杂 Bundle。字段存在于 Container 不等于内部模块可以直接依赖 Container。

### Step 2：迁移同步确定性装配

把 `create_app()` 中从 settings 到 orchestrator 的装配复制到 `build_app_container()`。

要求：

- 先保持行为一致，不做优化性重排。
- 保留 `_validate_real_tool_configuration(settings)`，可移动到 container 模块或继续从 main 引入，但最终不应让 container 反向依赖 FastAPI。
- 注意避免循环 import。
- `ApprovalService.orchestrator = orchestrator` 原样迁移并记录为已知残余耦合，本任务不重构审批恢复协议。

### Step 3：迁移 MCP startup

把 FastAPI lifespan 中 MCP 初始化逻辑移动到：

```python
await container.startup()
```

FastAPI lifespan 只调用 container 生命周期。

同时实现 `_started` 幂等保护，并补充 startup 失败时不置 started 的测试。

### Step 4：收敛 app.state 挂载

`create_app()` 中只挂载 container：

```python
app.state.container = container
```

挂载必须发生在 `create_app()` 返回前，而不是 lifespan startup 内。

### Step 5：收敛 main.py import

`app/main.py` 不应再直接 import 大量 runtime 构建细节。

目标上，`main.py` 只保留：

- FastAPI
- request/response schema
- auth dependency
- approval callback schema
- settings
- container builder
- RequestAdapter / ResponseAdapter
- HTTP 异常和日志

### Step 6：补充测试

新增或修改测试：

```text
tests/test_app_container.py
```

至少覆盖：

1. `build_app_container()` 能构建完整 runtime。
2. container 中存在 orchestrator、tool_registry、approval_service、verification_service。
3. public/private tools 已注册。
4. AgentCard 与 SkillCatalog 校验仍执行。
5. `container.startup()` 在 `ENABLE_MCP_CLIENT=false` 时可安全执行。
6. `container.startup()` 重复调用不会重复初始化或重复注册 MCP tools。
7. startup 失败时 `_started` 不得为 `True`。
8. 未 startup 调用 `container.shutdown()` 安全 no-op。
9. create_app() 返回时只存在 `app.state.container`，不再存在 `app.state.orchestrator`、`app.state.approval_store` 等平铺字段。
10. MCP startup 后动态注册的工具，已构建的 `ToolExecutor` / SubAgent 可见性逻辑能读到。
11. `/api/chat` smoke test 继续通过。

如果已有测试覆盖这些行为，可复用并补少量断言。

## 验收标准

- `app/main.py:create_app()` 不再直接承担 runtime 全量装配。
- 新增 `app/bootstrap/container.py`，并提供 `build_app_container()`。
- Container 生命周期是“同步确定性装配与本地静态初始化 + 异步 startup/shutdown”。
- AppContainer 只在入口、生命周期和集成测试中使用，内部业务组件不依赖整个 Container。
- FastAPI lifespan 只负责调用 container startup/shutdown。
- `app.state.container` 在 `create_app()` 返回前已挂载，且不再平铺挂载 runtime 组件。
- FastAPI routes 行为不变。
- MCP client 初始化和 MCP tools 注册行为不变。
- `startup()` 具备幂等语义，重复调用不重复初始化或重复注册。
- startup 失败时 `_started` 不得为 `True`。
- `shutdown()` 对未启动容器安全 no-op；当前无真实 close 接口时必须如实报告 no-op/有限清理。
- MCP 动态注册后，已构建的 ToolExecutor/SubAgent 能看到新工具。
- Tool contract 校验行为不变。
- Approval callback 和 resume 行为不变。
- `ApprovalService.orchestrator` 回填行为保持不变。
- 全量测试通过。

## 风险点

- `main.py` 与新 container 模块之间可能出现循环 import。
- 测试中不得继续直接依赖 `app.state.xxx` 平铺字段，应统一通过 `app.state.container` 访问 runtime 组件。
- MCP startup 如果重复调用，可能重复注册工具或覆盖状态，需要明确幂等策略。
- `ToolRegistry.register_mcp_tools()` 当前是同名覆盖式注册，但不会自动清理已消失的 MCP tool；本任务不要扩大为 MCP refresh 语义重构。
- `ApprovalService.orchestrator` 必须在 orchestrator 构建后回填。
- GraphFactory 构建依赖较多，迁移时不要改变参数。
- `sqlite_db_path` 测试覆写必须继续生效。
- `RequestAdapter` 当前调用 `get_settings()`，保留在 API 层时需避免与 container settings 产生配置漂移；本任务可先保持现状并在报告中记录。
- 当前 `MCPClientManager` 没有 close/shutdown 接口，shutdown 不得夸大为完整资源释放。
