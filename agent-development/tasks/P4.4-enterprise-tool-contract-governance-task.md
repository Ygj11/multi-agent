# P4.4 Enterprise Tool Contract Governance Task

优先级：企业级 P3

## 任务定位

本任务用于把当前工具体系从“注册函数并执行”升级为“受契约治理的企业能力”。

但本任务不追求一次性做全量平台化工具治理。当前项目阶段更需要的是：

- 工具不会无限卡死。
- 工具返回结构变化时能被发现。
- 工具异常有稳定错误码。
- 写工具审批边界明确。
- 新增工具必须声明最小契约。

因此 P4.4 调整为 **P4.4A：工具契约最小治理**。

高级能力如复杂 retry、幂等键执行、熔断器、复杂审批策略，不在本任务编码范围内，后置到 P4.4B 或更后续任务。

## 当前问题

### 1. 工具契约分散在注册代码里

当前 `app/tools/agent_tools.py`、POS 工具注册和 troubleshooting 工具注册里同时包含：

- 工具注册
- 参数 schema
- mock/real 分发
- 风险字段
- operation/is_write

长期会让工具注册脚本越来越大，也不利于统一检查每个工具的运行边界。

### 2. 工具缺少统一执行边界

当前 `ToolDefinition` 已有不少字段：

```text
name
parameters
is_write
operation
required_scopes
resource_type
risk_level
idempotency_required
data_classification
precondition_id
```

因此本任务不再堆叠大量重复字段，只补真正必要的执行边界：

- timeout
- result schema
- stable error code
- approval policy override
- minimal retry declaration

### 3. 工具结果没有统一 schema validation

现在工具执行后，结果会直接作为 observation 给 LLM。

如果工具返回结构变化，LLM 可能误读，最终回答也可能偏离真实工具结果。

### 4. 异常错误码不够稳定

当前普通异常可能直接使用 `str(exc)` 作为 error。

这不利于：

- 监控聚合
- 排障定位
- 自动恢复
- 用户友好提示
- 审计分类

### 5. 写工具审批策略仍主要依赖 `is_write`

当前写工具主要依赖 `is_write` 判断是否需要审批。

P4.4A 不引入复杂审批策略引擎，但需要明确：

- contract 中有 `approval_policy_id` 时优先使用。
- 没有 policy 时，`operation=write|notify` 或 `is_write=true` 默认需要审批。
- `operation=read|search` 默认不需要审批。

## 明确不做的过度设计

### 1. 不新增 `side_effect_type`

原因：当前已经有：

```python
operation: Literal["read", "write", "notify", "execute", "search"]
is_write: bool
```

如果再新增：

```python
side_effect_type
```

容易出现三套字段不一致：

```text
operation = read
side_effect_type = write
is_write = false
```

P4.4A 统一复用并强化 `operation`，不新增 `side_effect_type`。

### 2. 不新增 `circuit_breaker_key`

原因：本任务不实现真实熔断器后台。

如果只加字段而不执行熔断逻辑，会变成摆设，后续还容易误导维护者。

熔断器放到后续 P4.4B 或生产稳定性专项中再设计。

### 3. 不做复杂 retry

P4.4A 只保留最小 retry declaration：

```yaml
retry:
  max_attempts: 1
```

默认不重试。

真实 retry 只在后续明确安全边界后再做，例如：

- read 工具 timeout
- transient HTTP error
- MCP timeout

写工具默认不重试，除非后续有明确幂等策略。

## 架构目标

新增最小 ToolContract：

```yaml
version: "1.0.0"
tools:
  query_endo_task_record:
    timeout_ms: 5000
    retry:
      max_attempts: 1
    result_schema: QueryEndoTaskRecordResult
    approval_policy_id: null
    idempotency_key_fields: []
    data_classification: internal
```

ToolExecutionPipeline 执行顺序升级为：

```text
availability
argument contract
authorization
pre-tool verification
approval policy
timeout execution
result schema validation
evidence/log
```

P4.4A 不改变业务工具返回含义，只增加边界校验和错误稳定性。

## 实施步骤

### 1. 定义最小 ToolContract 模型

新增：

```python
class RetryPolicy(BaseModel):
    max_attempts: int = 1


class ToolContract(BaseModel):
    tool_name: str
    timeout_ms: int = 10000
    retry: RetryPolicy = RetryPolicy()
    result_schema: str | None = None
    approval_policy_id: str | None = None
    idempotency_key_fields: list[str] = []
    data_classification: Literal["public", "internal", "confidential", "sensitive"] = "internal"
```

说明：

- `tool_name` 是契约主键。
- `timeout_ms` 是 P4.4A 必须生效字段。
- `result_schema` 是 P4.4A 必须生效字段。
- `retry.max_attempts` 第一阶段默认保持 `1`，不做复杂 retry。
- `approval_policy_id` 第一阶段只作为审批策略 override。
- `idempotency_key_fields` 第一阶段只声明，不强制实现幂等执行。
- 不新增 `side_effect_type`，直接复用 `ToolDefinition.operation`。
- 不新增 `circuit_breaker_key`。

### 2. 新增 `tool_contracts.yaml`

先覆盖当前已注册工具：

- troubleshooting private tools
- pos_query private tools
- public tools
- MCP tools 可先支持通配或默认 contract

建议结构：

```yaml
version: "1.0.0"
defaults:
  timeout_ms: 10000
  retry:
    max_attempts: 1
  data_classification: internal
tools:
  query_task_status:
    timeout_ms: 5000
    result_schema: QueryTaskStatusResult
  notice_policy_update:
    timeout_ms: 5000
    result_schema: NoticePolicyUpdateResult
    approval_policy_id: write_tool_default
    idempotency_key_fields:
      - apply_seq
      - policy_no
```

### 3. ToolRegistry 合并 contract

工具注册 API 保持兼容：

- `register_private`
- `register_public`
- `register_mcp_tools`

Registry 加载 contract 后，把 contract 合并到 `ToolDefinition`。

建议优先方式：

```python
class ToolDefinition(BaseModel):
    ...
    contract: ToolContract | None = None
```

如果为了日志或测试便利需要扁平字段，可以通过只读属性或 helper 读取：

```python
definition.contract.timeout_ms
definition.contract.result_schema
```

不建议把所有 contract 字段复制成 `ToolDefinition` 顶层字段，避免模型继续膨胀。

缺 contract 策略：

- local/test：允许启动，但测试需要能发现缺失。
- prod：缺 contract 启动失败。

### 4. 增加 contract 覆盖测试

测试内容：

- 所有注册工具都有 contract。
- contract 中不存在未知工具。
- contract 中 result_schema 名称存在于 registry。
- prod 模式缺 contract 应失败。
- local/test 模式可以 warning 或允许继续。

### 5. 工具执行增加 timeout

在 `ToolExecutor._execute_definition()` 或 `ToolExecutionPipeline` 中对真实调用包裹 timeout。

超时返回稳定错误：

```text
tool_timeout
```

要求：

- local tool timeout 返回 `tool_timeout`
- MCP timeout 仍可保留 `mcp_tool_timeout`
- timeout 结果必须 `success=False`
- timeout 结果不得作为成功 observation 给 LLM

### 6. 增加 result schema validation

新增 result schema 注册表。

工具执行成功后：

```text
raw result
-> result schema validation
-> ToolResult(success=true)
```

如果 schema validation 失败：

```text
ToolResult(
  success=False,
  error="tool_result_schema_invalid"
)
```

要求：

- schema 失败不应把原始错误结构当成成功 observation 给 LLM。
- result schema 只校验结构，不改变业务含义。
- schema validation 错误详情可进入 `result.detail`，但 `error` 必须是稳定错误码。

### 7. 稳定错误码映射

新增 `app/tools/error_codes.py`：

```text
tool_not_found
tool_not_available_for_agent
missing_required_argument
permission_denied
verification_failed
human_approval_required
tool_timeout
tool_execution_exception
tool_result_schema_invalid
mcp_server_unavailable
mcp_tool_timeout
mcp_tool_error
```

P4.4A 必须做到：

- 普通异常不再直接暴露 `str(exc)` 作为 `ToolResult.error`。
- `ToolResult.error` 使用稳定错误码。
- 异常详情可以放入 `ToolResult.result.detail`。

### 8. 审批策略接入最小 contract

当前主要依赖 `is_write` 判断审批。

P4.4A 调整为：

```text
if approval_policy_id:
    use approval policy id
elif definition.is_write or operation in {"write", "notify"}:
    require approval
else:
    no approval
```

本任务不实现复杂 policy engine。

`approval_policy_id` 第一阶段可以只作为 metadata 写入 approval payload，便于后续扩展。

### 9. 保持 retry 为声明字段

本任务只解析并校验：

```yaml
retry:
  max_attempts: 1
```

不实现复杂 retry 执行。

如果未来要实现 retry，需要新任务明确：

- 哪些错误可重试
- read/write 是否允许重试
- 幂等键如何生成
- retry attempt 如何进入 tool log

### 10. 增加测试

测试覆盖：

- 所有注册工具都有 contract。
- prod 环境缺 contract 启动失败。
- 工具超时返回 `tool_timeout`。
- 工具结果 schema 不匹配返回 `tool_result_schema_invalid`。
- 普通异常返回 `tool_execution_exception`，异常详情不作为 error code。
- 写工具无审批 policy 时默认仍需审批。
- read 工具不触发审批。
- contract 中不允许未知 result_schema。

## 不在本任务范围

- 不实现真实熔断器。
- 不新增 `side_effect_type`。
- 不新增 `circuit_breaker_key`。
- 不实现复杂 retry。
- 不强制实现 idempotency key 执行逻辑。
- 不做可视化工具管理页面。
- 不改变业务工具返回数据含义。
- 不做 prompt eval。

## 验收标准

- `app/tools/tool_contracts.yaml` 存在并覆盖当前注册工具。
- `ToolContract` 模型存在，字段保持最小必要。
- `ToolDefinition` 能关联 contract。
- `ToolExecutionPipeline` 或 `ToolExecutor` 执行 timeout。
- 工具成功结果能按 `result_schema` 做 schema validation。
- schema validation 失败返回 `tool_result_schema_invalid`。
- 普通异常映射为 `tool_execution_exception`。
- 写工具审批策略由 contract 或默认规则明确控制。
- 新增测试全部通过。
