# P1.5 任务：落地 Tool Precondition Verification

## 任务背景

当前项目已经具备工具执行前验证的框架入口：

```text
ToolCallingRunner
  -> ToolExecutor.execute
  -> ToolExecutionPipeline.run
  -> ToolVerificationGuard
  -> ToolExecutor._verify_pre_tool
  -> VerificationService.verify(stage="pre_tool")
```

同时 `ToolDefinition` 已经预留：

```python
precondition_id: str | None = None
```

但当前缺口是：`precondition_id` 还没有被真正消费，项目也没有业务级 `PreconditionVerifier`。因此 LLM 在 tool loop 中可能跳过必要的前置查询，直接调用后续业务处理工具。虽然写工具会被审批拦截，但它仍然可能创建一个“证据不足”的审批单。

本任务目标是补齐：

```text
ToolDefinition.precondition_id
  -> VerificationService pre_tool
  -> ToolPreconditionVerifier
  -> 基于 evidence/tool history 校验业务前置条件
```

---

## 当前状态

### 已有能力

- `app/tools/execution_pipeline.py::ToolExecutionPipeline.run` 已有 `verification_guard`。
- `app/tools/executor.py::ToolExecutor._verify_pre_tool` 已调用 `VerificationService.verify(stage="pre_tool")`。
- `app/tools/base.py::ToolDefinition` 已有 `precondition_id` 字段。
- `app/verification/schemas.py` 已支持 `VerificationStage = "pre_tool"`。
- `ToolCallingRunner.run(...)` 会把已执行工具结果作为 `evidence` 传给 `ToolExecutor.execute(...)`。

### 缺失能力

- 没有 `ToolPreconditionVerifier`。
- `VerificationInput` 当前没有显式携带 `precondition_id`。
- `ToolExecutor._verify_pre_tool` 没有把当前 `ToolDefinition.precondition_id` 传入 verifier。
- `register_agent_private_tools(...)` 中的高风险业务处理工具没有配置 `precondition_id`。
- 没有 `precondition_id -> 规则` 的集中映射。
- 没有测试覆盖“LLM 直接调用处理工具时被阻断，先查询证据后才允许”的行为。

---

## 目标

实现工具执行前的业务前置条件校验，避免 LLM 直接调用后续处理工具。

以保全任务完成后异常处理为第一批落地场景：

| 工具 | 前置条件 |
|---|---|
| `notice_policy_update` | 已成功调用 `query_endo_task_record`，且 evidence 中存在 `task_type=9`、`task_status=E`、`response_body` 包含 `保单更新错误` |
| `notice_customer_update` | 已成功调用 `query_endo_task_record`，且 evidence 中存在 `task_type=9`、`task_status=E`、`response_body` 包含 `调用新客户接口异常` |
| `notice_period_update` | 已成功调用 `query_endo_task_record`，且 evidence 中存在 `task_type=9`、`task_status=E`、`response_body` 包含 `账单更新异常，失败` |
| `notice_finance` | 已成功调用 `query_endo_task_record`，且 evidence 中存在 `task_type=10`、`task_status=E` |
| `policy_suspendOrRecovery` | 已成功调用 `query_endo_task_record`，且 evidence 中存在 `task_type=11`、`task_status=E` |

---

## 设计原则

1. 不把前置条件只写在 prompt / SKILL.md 中。
2. 不相信 LLM 自觉遵守 SOP。
3. 不绕过 ToolExecutor。
4. 不绕过 VerificationService。
5. 不替代 AuthorizationService，权限校验仍然独立。
6. 不替代人工审批，写工具仍然需要审批。
7. precondition 不直接执行工具，只检查已有 evidence 是否满足条件。
8. precondition 失败应返回可控 `ToolResult`，作为 observation 回给 LLM。
9. 首期规则可以代码化，但结构要可迁移到配置化。

## 详细实现步骤

### 1. 扩展 VerificationInput

文件：

`app/verification/schemas.py`

为 `VerificationInput` 增加可选字段：

```python
precondition_id: str | None = None
tool_definition_metadata: dict[str, Any] = Field(default_factory=dict)
```

建议同时保留：

```python
tool_name
tool_arguments
evidence
agent_name
principal
auth_context
```

验收：

- 现有 `VerificationInput` 构造不受影响。
- 未传 `precondition_id` 时 verifier 应直接 pass。

---

### 2. ToolExecutor._verify_pre_tool 传入 precondition_id

文件：

`app/tools/executor.py`

当前 `_verify_pre_tool(...)` 只知道 `tool_name/arguments/evidence`。需要在调用前取出 tool definition：

```python
definition = self.registry.get_definition(tool_name)
precondition_id = definition.precondition_id if definition else None
```

然后传给：

```python
VerificationInput(
    stage="pre_tool",
    precondition_id=precondition_id,
    tool_definition_metadata=definition.metadata if definition else {},
    ...
)
```

验收：

- `ToolPreconditionVerifier` 能拿到当前工具的 `precondition_id`。
- 没有 definition 时仍然按原有逻辑返回工具不存在，不进入 verifier。

---

### 3. 新增 ToolPreconditionVerifier

新增文件：

`app/verification/verifiers/tool_precondition_verifier.py`

建议结构：

```python
class ToolPreconditionVerifier:
    name = "tool_precondition_verifier"
    stages = {"pre_tool"}

    async def verify(self, input: VerificationInput) -> VerificationResult:
        if not input.precondition_id:
            return pass
        rule = self.rules.get(input.precondition_id)
        if rule is None:
            return block or pass?
        return rule(input)
```

建议首期 fail-closed：

```text
precondition_id 存在但找不到规则 -> block
```

原因：工具显式声明了前置条件，系统却不知道如何验证，应该阻断而不是放行。

验收：

- 无 `precondition_id`：pass。
- 有未知 `precondition_id`：block，code=`precondition_rule_not_found`。
- 有已知 `precondition_id` 且 evidence 满足：pass。
- 有已知 `precondition_id` 但 evidence 不满足：block。

---

### 4. 实现 aftercare 规则

首期内置规则：

```text
endo_aftercare.policy_update_failed
endo_aftercare.customer_update_failed
endo_aftercare.period_update_failed
endo_aftercare.finance_failed
endo_aftercare.recovery_failed
```

规则检查 evidence 中是否存在对应 `query_endo_task_record` 的成功结果。

需要兼容 `ToolCallingRunner` 当前 evidence 结构：

```python
executed_calls.append(tool_result.model_dump())
```

所以 evidence item 可能长这样：

```json
{
  "name": "query_endo_task_record",
  "success": true,
  "result": {
    "apply_seq": "APPLY_POLICY_UPDATE_FAIL",
    "records": [
      {
        "task_type": "9",
        "task_status": "E",
        "response_body": "保单更新错误：xxx"
      }
    ]
  }
}
```

检查 helper 建议：

```python
def _has_endo_record(input, *, task_type, status="E", response_contains=None) -> bool:
    ...
```

验收：

- 可以从 `result.records` 找到匹配节点。
- `response_body` 包含目标关键字时才通过。
- 不应因为 evidence 中有无关工具结果而通过。

---

### 5. 注册 ToolPreconditionVerifier

文件：

`app/bootstrap/verification.py`

当前可能类似：

```python
return VerificationService(
    [
        DataPermissionVerifier(),
        ComplianceVerifier(llm_provider=llm_provider),
    ]
)
```

改为：

```python
return VerificationService(
    [
        DataPermissionVerifier(),
        ToolPreconditionVerifier(),
        ComplianceVerifier(llm_provider=llm_provider),
    ]
)
```

顺序建议：

```text
DataPermissionVerifier
ToolPreconditionVerifier
ComplianceVerifier
```

说明：

- 数据权限是确定性安全边界。
- precondition 是业务执行边界。
- compliance 更多用于内容安全和最终输出。

验收：

- `VerificationService.verify(stage="pre_tool")` 会运行 `ToolPreconditionVerifier`。

---

### 6. 给 aftercare 处理工具配置 precondition_id

文件：

`app/tools/agent_tools.py`

给工具注册增加：

```python
notice_policy_update:
  precondition_id="endo_aftercare.policy_update_failed"

notice_customer_update:
  precondition_id="endo_aftercare.customer_update_failed"

notice_period_update:
  precondition_id="endo_aftercare.period_update_failed"

notice_finance:
  precondition_id="endo_aftercare.finance_failed"

policy_suspendOrRecovery:
  precondition_id="endo_aftercare.recovery_failed"
```

注意：

- `query_endo_task_record` 不需要 precondition。
- precondition 失败时不应进入人工审批。
- precondition 通过后，如果工具是 `is_write=True`，仍然进入审批。

验收：

- LLM 直接调用 `notice_policy_update`，没有 evidence 时返回 `verification_failed:precondition_not_satisfied`。
- 先调用 `query_endo_task_record` 得到 9 节点失败证据后，再调用 `notice_policy_update` 才能通过 precondition。

## 验收标准

1. `VerificationService` 真正消费 `precondition_id`。
2. `ToolDefinition.precondition_id` 不再只是预留字段。
3. aftercare 处理类工具不能在缺少前置查询证据时被执行或进入审批。
4. 先查询并获得匹配证据后，后续工具可以继续走审批/执行链路。
5. precondition 失败以 `ToolResult` 形式返回，不抛未捕获异常。
6. precondition 失败结果会作为 tool observation 回给 LLM。
7. 不改变 AgentGraph 主链路。
8. 不绕过 AuthorizationService。
9. 不绕过 is_write 人工审批。
10. `uv run python -m compileall app tests` 通过。
11. `uv run pytest` 通过。

---

## 风险点

1. evidence 结构当前来自 `ToolResult.model_dump()`，后续如果 ToolResult 字段变化，precondition helper 需要同步。
2. 规则写死在 verifier 中只是首期方案，后续可迁移到配置文件。
3. 如果工具结果很长，evidence 中 raw result 会膨胀；后续应配合 observation summary / evidence store。
4. 如果某个工具的 `is_write=True`，precondition 通过不代表可以执行，只代表可以进入下一道审批边界。
5. 未知 `precondition_id` 采用 fail-closed 可能影响新工具上线，需要测试和配置流程兜底。
