# P5.7 Architecture Comment Governance Task

优先级：企业级 P5.7

## 任务定位

当前项目已经完成多轮主干治理：Query Rewrite 上下文继承、EntityResolver 实体收敛、Intent Taxonomy、Agent/Skill/Tool 边界、Tool Contract、Runtime Container、状态瘦身和历史残留清理。

这些能力让架构边界更清晰，但也带来一个新问题：很多关键设计约定只存在于代码行为和历史任务讨论中，源码内的架构注释偏少。后续开发者如果只读代码，很容易把以下边界改偏：

- `entity_bag` 与 `entities` 的事实源关系。
- Query Rewrite、Intent Recognition、Agent Selection、Skill Selection 的职责边界。
- LLM 输出、规则 fallback、EntityResolver 的优先级关系。
- ToolDefinition、ToolContract、ToolRegistry、ToolExecutor 的分层关系。
- Approval、Verification、Authorization 的安全边界。
- Runtime Container、HTTP Client、MCP Client 的生命周期归属。

本任务用于补强源码和 YAML 中的“架构型注释”，不是补普通逐行注释，也不是新增文档。

## 本任务不修改 docs

本任务明确不修改：

- `docs/`
- `opt/`
- `revision-notes/`
- `README.md`

需要新增或修改的注释只允许出现在：

- Python 源码中的模块、类、函数、关键分支附近；
- YAML 配置文件中的字段语义注释；
- 必要时在测试中补少量注释说明断言意图。

任务说明文件本身位于 `tasks/`，用于指导后续开发，不属于运行时文档。

## 总体目标

### 目标 1：补足架构边界注释

在关键模块中解释“为什么这样设计”和“不可破坏的约定”，而不是解释每行代码做什么。

注释应回答：

```text
这个模块/函数的职责边界是什么？
它不能承担哪些职责？
它和上游/下游的契约是什么？
哪些字段是 canonical state，哪些只是兼容投影？
哪些 fallback 是业务允许的，哪些只是本地开发兜底？
```

### 目标 2：补足配置语义注释

在关键 YAML 中说明字段被谁消费、配置为空时意味着什么、配置存在时会触发什么校验。

重点说明：

```text
配置了就校验
未配置就不施加该项约束
```

但不能为了注释而改变现有配置语义。

### 目标 3：避免注释漂移

所有注释必须来自当前真实源码行为，不能把目标架构、未来规划或尚未实现能力写成当前事实。

如果发现代码和既有 README / docs / 任务文档不一致，以当前运行时代码为准。

## 注释原则

### 1. 写架构注释，不写噪声注释

推荐：

```python
# IntentRecognitionNode 只负责识别 intent/sub_intent。
# canonical 实体状态已经由 QueryRewriteNode + EntityResolver 形成，
# 本节点不能再把 LLM 返回的 entities 写回 entity_bag。
```

避免：

```python
# 遍历列表
for item in items:
    ...
```

### 2. 注释放在事实来源附近

```text
状态事实源约定 -> 写在 state 更新 helper / EntityBag / graph 节点附近
实体合并优先级 -> 写在 EntityResolver 附近
字段配置语义 -> 写在对应 YAML 字段附近
工具执行安全边界 -> 写在 ToolExecutor / ExecutionPipeline 附近
资源生命周期 -> 写在 AppContainer / client lifecycle 附近
```

### 3. 不把 README 搬进代码

代码注释只说明当前模块必须知道的边界。

全局架构图、完整请求链路、开发指南仍由 README 或后续文档承载，本任务不处理。

### 4. 注释必须跟随当前代码

禁止出现：

- 当前未实现但注释说“已支持”；
- 当前只是 fallback，但注释说“主流程强依赖”；
- 当前只是本地开发能力，但注释没有标明环境边界；
- 注释描述旧字段、旧节点或已删除 agent/skill。

## P5.7.2 Query Rewrite 与实体解析注释

### 目标文件

- `app/query/query_rewrite_node.py`
- `app/query/entity_resolver.py`
- `app/query/entity_extractor.py`
- `app/query/entity_patterns.yaml`
- `app/schemas/entities.py`
- `app/schemas/query_rewrite.py`

### 需要补充

1. Query Rewrite 的边界：
   - 不是简单润色 query；
   - 负责把依赖上下文的问题改写成自包含问题；
   - 负责当前轮实体、历史候选、LLM 候选进入 EntityResolver 前的编排。

2. LLM 实体的定位：

   ```text
   LLM 返回的 entities 是候选，不是 canonical entity state。
   只有经过 EntityResolver 后，才能进入 entity_bag。
   ```

3. 规则 fallback 的定位：
   - LLM 不可用、输出非法或置信不足时使用；
   - fallback 仍必须通过统一实体状态更新；
   - fallback 不能绕过 EntityResolver 私自合并实体。

4. EntityResolver 的覆盖优先级：
   - 当前轮明确实体优先于历史实体；
   - 当前轮确定性实体优先于 LLM 候选；
   - 历史实体只有唯一高置信时才可继承；
   - 同优先级冲突要澄清；
   - 当前轮同类型多值可以保留为集合语义。

5. `regex` 与 `value_regex` 的区别：
   - `regex` 用于从原始文本抽取；
   - `value_regex` 用于校验已经被拆出来的值；
   - 二者不能随意混用。

### 不允许

- 不新增 Entity Agent。
- 不新增 ToolParameterBinder。
- 不改变实体抽取行为。

## P5.7.3 Intent Recognition 与路由注释

### 目标文件

- `app/query/intent_recognition_node.py`
- `app/query/intent_taxonomy.py`
- `app/config/intent_taxonomy.yaml`
- `app/agents/selection.py`
- `app/agents/cards/*.yaml`

### 需要补充

1. Intent Recognition 的边界：
   - 只识别 `intent/sub_intent`；
   - 不选择 Agent；
   - 不选择 Skill；
   - 不修改 canonical entity state。

2. Taxonomy、AgentCard、Skill 的关系：

   ```text
   intent_taxonomy.yaml 定义系统认识的业务意图值域。
   AgentCard supported_routes 声明哪个 Agent 能处理哪些 route。
   Skill frontmatter 声明该 Skill 服务的 intent/sub_intent。
   ```

3. AgentSelection 打分来源：
   - taxonomy 合法性；
   - supported_routes；
   - required_entities；
   - capabilities；
   - examples；
   - keywords；
   - enabled 状态；
   - 必要时 LLM router 只在候选中选择。

4. fallback 规则：
   - LLM router 不可用时回退规则选择；
   - 规则选择不是“随便猜”，仍基于候选约束。

### 不允许

- 不改变 taxonomy。
- 不修改 AgentCard 业务字段。
- 不新增 Agent Selection 策略。

## P5.7.4 Skill 与 SubAgent 注释

4. SubAgent Context 的来源：

### 不允许

## P5.7.5 Tool Contract 与执行安全注释

### 目标文件

- `app/tools/registry.py`
- `app/tools/contracts.py`
- `app/tools/executor.py`
- `app/tools/execution_pipeline.py`
- `app/tools/agent_tools.py`
- `app/config/tool_contracts.yaml`

### 需要补充

1. ToolDefinition 与 ToolContract 的关系：
   - ToolDefinition 是运行时可执行工具定义；
   - ToolContract 是治理补丁/静态契约；
   - 注册时可以用 contract 覆盖 operation、risk_level、data_classification 等治理字段。

2. ToolRegistry 的职责：
   - 注册 local/public/private/admin/MCP tools；
   - 维护 agent 可见性；
   - 向 LLM 输出 function calling schema；
   - 不负责真正执行工具。

3. ToolExecutor / ExecutionPipeline 的职责：
   - 工具存在性校验；
   - Agent 可见性校验；
   - 参数 schema/required 校验；
   - 权限校验；
   - precondition/approval/idempotency；
   - 执行工具；
   - 结果过滤和日志。

4. LLM 的边界：

   ```text
   LLM 只能提出 tool call。
   ToolExecutor 才是工具是否真正执行的安全边界。
   ```

5. MCP 未知工具策略：
   - 按当前 `UNKNOWN_MCP_TOOL_POLICY` 或真实配置说明；
   - 不把未知 MCP 工具默认描述成安全。

### 不允许

- 不改工具注册逻辑。
- 不改工具 schema。
- 不改审批策略。

## P5.7.6 Auth / Approval / Verification 注释

### 目标文件

- `app/auth/`
- `app/approval/`
- `app/verification/`
- `app/api/`

### 需要补充

1. 身份来源边界：
   - Header / trusted principal；
   - body fallback 只用于本地开发；
   - `APP_ENV=prod`、`AUTH_MODE=required`、`ALLOW_REQUEST_BODY_IDENTITY_FALLBACK=false` 时必须使用可信身份。

2. Authorization 与 Verification 的区别：
   - Authorization 是执行前能不能做；
   - Verification 是回答前是否合规、可返回。

3. Approval 的边界：
   - 高风险/写操作可能创建 approval request；
   - 当前恢复机制基于 ApprovalStore 保存 resume state；
   - 如果未使用 LangGraph 原生 interrupt，不要注释成原生 interrupt。

4. Verification 当前能力边界：
   - 哪些是强制校验；
   - 哪些只是固定 fallback/安全话术；
   - 不夸大成智能语义修复。

### 不允许

- 不修改认证策略。
- 不改变审批表结构。
- 不新增 verification 规则。

## P5.7.7 Runtime Container 与 Client 生命周期注释

### 目标文件

- `app/bootstrap/container.py`
- `app/runtime/async_client_lifecycle.py`
- `app/integrations/clients.py`
- `app/integrations/*`
- `app/mcp/*`

### 需要补充

1. Runtime Container 边界：
   - FastAPI 是入口；
   - AppContainer 是 runtime 装配中心；

2. Client 生命周期：
   - 谁创建；
   - 谁持有；

3. HTTP Client 连接池边界：
   - 连接复用不等于自动重试写操作；
   - 不允许运行时修改共享 client 的全局 headers/cookies；
   - 不同上游配置应使用独立 client。

### 不允许

- 不再重构 container。
- 不改 client shutdown 行为。
- 不新增连接池配置。

## 实施步骤

### 阶段一：注释现状审计

先检查关键模块，列出：

- 完全没有边界注释的核心类；
- 有注释但已经过期的位置；
- YAML 字段语义不清的位置；
- 容易被误改的关键分支；
- 注释中可能与当前实现不一致的位置。

### 阶段二：补代码架构注释

优先顺序：

1. `graph.py`
2. `query_rewrite_node.py`
3. `entity_resolver.py`
4. `intent_recognition_node.py`
5. `selection.py`
6. `base.py` / `runner.py`
7. `registry.py` / `execution_pipeline.py`
8. `container.py` / client lifecycle

### 阶段三：补 YAML 字段语义注释

优先顺序：

1. `entity_patterns.yaml`
2. `intent_taxonomy.yaml`
3. AgentCard YAML
4. Skill frontmatter
5. `tool_contracts.yaml`

### 阶段四：自检注释准确性

检查：

- 注释是否描述当前真实行为；
- 是否写入了未来规划；
- 是否有“生产级”“完整支持”等无法由代码证明的表述；
- 是否把 fallback 写成主流程；
- 是否把本地开发能力写成生产能力；
- 是否误导开发者绕过安全边界。
