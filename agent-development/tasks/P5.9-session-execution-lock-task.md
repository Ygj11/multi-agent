# P5.9 Session Execution Lock Task

## 0. 任务定位

本任务解决同一会话并发请求导致的语义竞态。

当前项目的 `thread_id` 由：

```python
f"{session_key}:{request_id}"
```

构成。同一 `session_key` 的两个请求如果同时进入系统，会进入不同 LangGraph thread，并并发读取/写入同一会话消息和短期记忆。

目标语义：

```text
跨 session 并行
同一 session 串行
```

本任务不删除 `request_id`，也不把所有请求挤到同一个 LangGraph thread，而是在 Orchestrator 入口增加 session 级异步锁。

```text
/api/approval/callback

```

### 1.4 当前竞态

如果同一 `session_key` 同时收到请求 A 和 B：

```text
A load_session 读取历史 H
B load_session 也读取历史 H
A query_rewrite 基于 H
B query_rewrite 也基于 H
B 先保存 assistant message
A 后保存 assistant message
short memory 压缩顺序不确定
```

在澄清、追问、实体继承场景中，这会造成明显语义错位。

## 2. 总体目标

新增：

```text
SessionExecutionLockManager
```

接入：

```text
AgentOrchestrator.run()
AgentOrchestrator.resume_after_approval()
```

目标：

```text
同一 session_key 同一时刻只允许一个 graph run。
不同 session_key 可以并发执行。
审批恢复和普通请求共享同一把 session lock。
```

## 3. 非目标

本任务不做：

- 不删除 `request_id`；
- 不把 `thread_id` 改成只使用 `session_key`；
- 不改 LangGraph 原生 checkpoint 语义；
- 不实现分布式锁；
- 不引入 Redis；
- 不改 API 请求协议；
- 不重写 Approval resume；
- 不解决多进程 / 多实例下的全局串行问题。

第一版只实现单进程内 session lock。

## 4. 为什么 request_id 仍然必要

`session_key` 和 `request_id` 是不同层次：

| 字段 | 作用 |
| --- | --- |
| `session_key` | 会话上下文边界，用于读取消息、短期记忆、串行锁 |
| `request_id` | 单次请求追踪 ID，用于日志、工具日志、审批、checkpoint、排错 |
| `thread_id` | Graph/checkpoint 执行线程 ID，目前是 `session_key:request_id` |

加 session lock 后：

```text
session_key 控制并发
request_id 控制可观测性和请求级隔离
thread_id 保持请求级 Graph run
```

因此 `request_id` 不应删除。

## 5. 设计方案

新增文件：

```text
app/runtime/session_locks.py
```

核心接口：

```python
class SessionExecutionLockManager:
    async def lock(self, session_key: str) -> AsyncIterator[SessionLockLease]:
        ...
```

或：

```python
async with session_locks.lock(session_key):
    state = await graph.ainvoke(...)
```

## 6. Lock 生命周期设计

### 6.1 进程内锁

第一版使用：

```text
dict[session_key, LockEntry]
asyncio.Lock
asyncio.Condition 或 guard lock
```

要求：

- 同一 session_key 复用同一把 lock；
- 不同 session_key 使用不同 lock；
- 请求结束后如果没有等待者，允许清理 lock entry；
- 避免 `_locks` 无限增长。

### 6.2 LockEntry 建议

```python
@dataclass
class SessionLockEntry:
    lock: asyncio.Lock
    active_count: int = 0
    waiter_count: int = 0
    last_used_at: float = 0.0
```

也可以更简单：

```text
guard lock 保护 dict
获取 entry 后 waiter_count +1
释放后 active/waiter 都为 0 时删除 entry
```

### 6.3 超时

新增配置：

```dotenv
SESSION_LOCK_TIMEOUT_SECONDS=60
SESSION_LOCK_MAX_ACTIVE=10000
```

第一版至少实现 timeout：

- 等待超过 timeout 后抛出 `SessionExecutionLockTimeout`;
- API 层最终返回明确错误；
- 或 Orchestrator 返回 state error：`session_busy_timeout`。

建议第一版在 Orchestrator 中捕获并返回业务可控状态，避免直接 500。

## 7. Orchestrator 接入

修改：

```text
app/runtime/orchestrator.py
app/bootstrap/container.py
```

构造：

```python
session_locks = SessionExecutionLockManager(
    timeout_seconds=settings.session_lock_timeout_seconds,
)
orchestrator = AgentOrchestrator(
    graph,
    checkpoint_store=storage.checkpoint_store,
    session_locks=session_locks,
)
```

### 7.1 普通请求

目标：

```python
async def run(self, inbound):
    async with self.session_locks.lock(inbound.session_key):
        return await self._run_without_session_lock(inbound)
```

### 7.2 审批恢复

目标：

```python
async def resume_after_approval(self, approval_request):
    session_key = approval_request.session_key or resume_contract.session_key
    async with self.session_locks.lock(session_key):
        return await self._resume_without_session_lock(approval_request)
```

审批 callback 和普通请求必须共享同一把 session lock。

## 8. 失败与返回策略

新增错误码建议：

```text
session_busy_timeout
session_lock_unavailable
```

超时响应建议：

```text
当前会话上一条请求仍在处理中，请稍后再试。
```

Graph state 可包含：

```text
error=session_busy_timeout
answer=...
graph_path=["session_lock_timeout"]
```

但要注意：

- 不要写入正常业务 assistant message，除非产品希望用户看到该提示；
- 是否落消息由最终设计决定；
- 如果超时发生在 graph 前，可能不进入 Graph path，需要 Orchestrator 层构造最小 state。

第一版建议：

```text
Orchestrator 抛出 SessionExecutionLockTimeout
FastAPI 转 409 Conflict
```

这样实现简单，语义清晰。

## 10. 多进程边界

第一版是进程内锁。

需要在 README / 任务报告中明确：

```text
单个 Uvicorn worker 内，同 session 串行。
多个 Uvicorn worker / 多实例部署时，不同进程之间不会共享这把锁。
生产多实例需要 Redis / DB advisory lock / queue 级 session partition。
```

后续可演进：

```text
DistributedSessionExecutionLock
Redis SET NX PX
数据库 session_locks 表
按 session_key partition 的消息队列
```

本任务不实现分布式锁。

## 11. Approval 兼容

必须验证：

1. 审批 pending 请求释放 session lock；
2. pending 期间新用户请求可以进入，还是需要阻塞？
3. 审批 callback resume 与普通请求同时到达时，同 session 串行；
4. callback resume 不会和新请求交叉写同一会话消息。

设计建议：

```text
Graph 执行到 pause_for_approval 并返回 pending 后，锁释放。
审批 pending 期间是否允许新请求进入，第一版允许，但 callback resume 会与它串行。
```

如果业务上希望 pending 期间拒绝新请求，需要另加会话状态机，本任务不做。

## 12. Checkpoint 与 thread_id 设计

当前保持：

```python
thread_id = f"{session_key}:{request_id}"
```

理由：

- request_id 级 snapshot 更容易排查；
- 当前 checkpoint store 保存最终状态快照，不是完整会话线程；
- session lock 已经解决同会话并发，不需要用 thread_id 承担锁职责。

禁止：

```text
不要为了串行，把 thread_id 直接改成 session_key。
```

否则可能造成：

- 不同请求 checkpoint 覆盖；
- 审批 resume 定位混乱；
- request 级 trace 变差。

## 15. 验收标准

必须满足：

1. 同一 `session_key` 的两个普通请求不会并发进入 `graph.ainvoke()`。
2. 不同 `session_key` 的请求可以并发。
3. 审批 callback resume 与普通请求共享同一把 session lock。
4. 请求异常时 lock 必须释放。
5. 等待超时时返回明确错误，不无限等待。
6. lock entry 不长期泄漏。
7. `request_id` 保留，thread_id 保持 `session_key:request_id`。
8. `uv run pytest -q` 通过。
9. `uv run python scripts/run_agent_evals.py` 通过。

## 16. 风险

| 风险 | 控制 |
| --- | --- |
| 单请求卡住导致同 session 后续请求阻塞 | lock timeout |
| lock dict 泄漏 | release 后无等待者即清理 |
| 多 worker 无法串行 | 明确第一版进程内锁，生产多实例另做分布式锁 |
| 审批 pending 期间业务语义复杂 | 第一版只保证执行时串行，不引入会话状态机 |
| 直接改 thread_id 造成 checkpoint 混乱 | 保持 request_id/thread_id 现状 |
| API 返回 500 | 显式异常类型并映射 409 |

## 17. 需要人工确认的问题

1. `ENABLE_SESSION_EXECUTION_LOCK` 是否生产环境强制 true。
2. session lock timeout 超时时 API 返回 `409` 还是返回一个正常 ChatResponse。
3. 审批 pending 期间是否允许用户继续发新请求。
4. 多 worker 部署是否需要立即规划 Redis/DB 分布式锁。
