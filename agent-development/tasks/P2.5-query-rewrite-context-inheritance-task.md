# P2.5 Query Rewrite Context Inheritance Task

## 背景

当前 `QueryRewriteNode` 同时支持 LLM 改写和规则兜底，用来把用户当前问题、历史消息、短期摘要和实体窗口整理成后续意图识别可以消费的请求。

现有规则兜底存在边界问题：

- 只有当前轮完全没有实体时，才会从历史继承实体。
- 如果当前轮有部分实体，例如只给了 `apply_seq`，规则不会继承上一轮唯一明确的 `policy_no`。
- 如果粗暴改成“历史有、当前没有就继承”，又会在多轮独立请求里误继承或误澄清。

典型反例：

```text
第 1 轮：保单号 9200100000458846
第 2 轮：保单号 9200100000458847
第 3 轮：受理号 930010412672222
```

第 3 轮出现了新的强锚点 `apply_seq`，它可能本身就是一个新请求，不应该因为历史里有两个 `policy_no` 就直接触发澄清，也不应该随便继承其中一个保单号。

因此，本任务要优化两部分：

- LLM query rewrite 提示词：明确什么时候可以继承历史实体，什么时候不能继承。
- 规则兜底逻辑：引入上下文引用判定和强锚点请求帧，避免粗暴合并。
- 澄清补参识别：区分“agent 上一轮要求补实体，用户本轮补参”和“agent 已回答后用户继续追问”。

## 目标

让 query rewrite 阶段做到：

- 当前轮实体优先。
- 历史实体只是候选，不自动合并。
- 只有用户明确引用上下文，或当前轮没有强锚点且是明显追问时，才考虑继承历史实体。
- 当前轮出现新的强锚点时，默认视为新请求帧，不继承历史实体。
- 多候选历史实体只有在用户明确选择时才定位，否则在确实需要继承时才澄清。
- `IntentRecognitionNode` 消费 query rewrite 整理好的实体，不承担历史实体继承职责。
- 明确区分 `clarification_reply` 和 `follow_up_question` 两种多轮语义。

## 多轮语义分类

多轮对话至少要区分两类：

### 1. 澄清补参 Clarification Reply

上一轮是系统/agent 发起澄清，要求用户补充缺失实体；本轮用户提供的内容应被视为对上一轮任务的补参。

示例：

```text
第 1 轮用户：保单号 9200100000458846 保全任务完成但没更新
第 1 轮 assistant：执行保全任务完成后异常处理还缺少保全项 endorseType，请补充后我再继续处理。
第 2 轮用户：001028
```

第 2 轮必须合并成：

```json
{
  "policy_no": "9200100000458846",
  "endorseType": "001028"
}
```

这里即使当前轮 `001028` 是强锚点或当前有实体，也不能把它当成新请求。因为上一轮系统明确在等待补参。

### 2. 普通追问 Follow-up Question

上一轮 agent 已经给出业务回答，用户继续追问原因、责任方、下一步等。

示例：

```text
第 1 轮用户：REQ_001 为什么 E102？
第 1 轮 assistant：初步判断是签名校验失败...
第 2 轮用户：那这个一般是谁的问题？
```

第 2 轮应继承上一轮上下文：

```json
{
  "request_id": "REQ_001",
  "error_code": "E102"
}
```

### 3. 新请求 New Request

当前轮出现新的强锚点，且上一轮不是 pending clarification，也没有明确上下文引用词，则默认是新请求。

示例：

```text
第 1 轮用户：保单号 9200100000458846
第 2 轮用户：保单号 9200100000458847
第 3 轮用户：受理号 930010412672222 查一下
```

第 3 轮不应继承第 1/2 轮保单号，也不应因为历史多个保单号而澄清。

## 关键概念

### 强锚点实体

强锚点表示当前轮足以开启一个新的请求帧。

当前建议包括：

```text
policy_no
apply_seq
request_id
task_id
claim_no
```

项目目前已经支持的实体优先使用：

```text
policy_no
apply_seq
request_id
task_id
```

### 上下文引用词

显式上下文引用：

```text
这个
那个
刚才
上一轮
前面
继续
接着
它
还是这个
刚才那个
```

序数/选择引用：

```text
第一个
第二个
上一个
下一个
前一个
后一个
上面那个
下面那个
```

弱追问语义：

```text
为什么
谁的问题
怎么办
怎么处理
状态呢
有没有更新
查一下
```

弱追问只有在当前轮没有强锚点时，才应判定为引用上下文。

## 设计原则

### 1. QueryRewrite 必须先判断多轮类型

`QueryRewriteNode` 的规则兜底和 LLM prompt 都应该先判断本轮属于哪类：

```text
clarification_reply
follow_up_question
new_request
direct_standalone
ambiguous
```

优先级：

```text
pending clarification > 显式上下文引用 > 无强锚点追问 > 新强锚点请求 > 普通独立请求
```

其中 `pending clarification` 优先级最高。

### 2. QueryRewrite 只整理请求，不判断业务必填是否完整

`QueryRewriteNode` 负责：

- 抽取当前轮实体。
- 判断是否引用上下文。
- 在允许时继承历史实体。
- 生成 `rewritten_query`、`entities`、`inherited_entities`、`entity_bag`、`conversation_window`。

它不负责判断某个 skill/tool 的实体是否足够。

业务必填实体仍由：

```text
SkillContextResolver -> RequiredEntityChecker
```

处理。

### 3. Pending Clarification 不是普通上下文引用

agent 是无状态的，但系统不是无状态的。上一轮 assistant 消息会进入 `recent_messages`，且 message metadata 可以保存：

```text
need_clarification
clarification_source
clarification_question
missing_required_entities
entities
intent
sub_intent
selected_agent
selected_skill_id
```

本任务要求把澄清相关 metadata 作为判定依据。

如果最近一条 assistant 消息满足：

```text
role = assistant
metadata.need_clarification = true
```

则下一轮用户消息优先按 `clarification_reply` 处理。

处理方式：

- 继承上一轮任务实体。
- 抽取当前轮补充实体。
- 当前轮实体覆盖同类型历史实体。
- 如果当前轮实体能够填补 `missing_required_entities`，则合并后继续原任务。
- 如果仍缺实体，继续澄清。

### 4. 当前强锚点默认开启新请求

例如：

```text
受理号 930010412672222 查一下
```

如果当前轮没有上下文引用词，则应输出：

```json
{
  "entities": {
    "apply_seq": "930010412672222"
  },
  "inherited_entities": {},
  "need_clarification": false
}
```

不要继承历史 `policy_no`。

例外：

- 上一轮是 pending clarification。
- 当前轮显式说“刚才/这个/第二个/继续”等上下文引用。

### 5. 明确引用上下文才继承

例如：

```text
刚才那个保单，受理号 930010412672222 查一下
```

如果历史 active frame 唯一可定位到某个 `policy_no`，可以继承：

```json
{
  "entities": {
    "policy_no": "9200100000458846",
    "apply_seq": "930010412672222"
  },
  "inherited_entities": {
    "policy_no": "9200100000458846"
  }
}
```

### 6. 多候选只有在需要继承且无法定位时才澄清

例如：

```text
第 1 轮：保单号 A
第 2 轮：保单号 B
第 3 轮：刚才那个继续查
```

如果无法知道“刚才那个”指 A 还是 B，应澄清。

但：

```text
第 3 轮：受理号 C 查一下
```

不应因为历史有 A/B 就澄清。

## LLM Prompt 优化

修改文件：

```text
app/prompts/query_rewrite/system.md
```

需要增加上下文继承规则：

```md
## Context Inheritance Rules

- Treat conversation_window entities as candidates, not values to always merge.
- Current-turn entities have priority over historical entities.
- Strong anchors start a new request frame by default.
- Strong anchors include policy_no, apply_seq, request_id, task_id, claim_no.
- If the current message contains a strong anchor and does not explicitly reference prior context, do not inherit historical entities.
- Explicit context references include: 这个, 那个, 刚才, 上一轮, 前面, 继续, 接着, 它, 第一个, 第二个, 上一个, 前一个, 后一个.
- Follow-up questions such as 谁的问题, 为什么, 怎么办, 状态呢 only count as context references when the current message has no strong anchor.
- If multiple historical candidates match and the user does not identify one, set need_clarification=true only when inheritance is required.
- Distinguish clarification_reply from follow_up_question.
- If the last assistant message has metadata.need_clarification=true, treat the current user message as a clarification reply first.
- For clarification_reply, inherit the previous task entities and merge current-turn entities to fill missing_required_entities.
- Do not classify a clarification reply as a new request merely because the current message contains a strong anchor.
- Put only inherited values in inherited_entities.
- Put current-turn entities plus valid inherited entities in entities.
```

同时检查：

```text
app/prompts/query_rewrite/user.md
```

确保 prompt 中保留：

- `original_query`
- `current_entities`
- `conversation_window`

## 规则兜底优化

修改文件：

```text
app/query/query_rewrite_node.py
```

建议新增小对象或轻量 dict：

```python
ConversationTurnType:
    value: Literal[
        "clarification_reply",
        "follow_up_question",
        "new_request",
        "direct_standalone",
        "ambiguous",
    ]
    reason: str

ContextReferenceResult:
    is_reference: bool
    reason: str
    target_hint: str | None
```

建议新增方法：

```python
def _detect_pending_clarification(
    self,
    recent_messages: list[dict[str, Any]],
) -> dict[str, Any] | None:
    ...
```

判定逻辑：

- 从最近消息倒序查找最近一条 assistant 消息。
- 如果 metadata.need_clarification 为 true，则返回该消息的 metadata。
- 该 metadata 至少要能提供上一轮 `entities`。
- 最好同时保存 `missing_required_entities`、`clarification_question`、`intent`、`sub_intent`、`selected_agent`、`selected_skill_id`。

需要同步检查：

```text
app/runtime/handlers/message_commit_handler.py
```

当前 assistant message metadata 已保存：

```text
need_clarification
clarification_source
entities
intent
sub_intent
selected_agent
```

本任务建议补充保存：

```text
missing_required_entities
clarification_question
selected_skill_id
selected_skill_metadata
```

否则下一轮很难精确知道用户补的是哪个缺失实体。

建议新增方法：

```python
def _detect_context_reference(
    self,
    query: str,
    current_bag: EntityBag,
) -> ContextReferenceResult:
    ...
```

判定优先级：

```text
pending clarification -> turn_type=clarification_reply
显式引用词 -> is_reference=true
序数/选择词 -> is_reference=true
弱追问且无强锚点 -> is_reference=true
有强锚点且无显式引用 -> is_reference=false
其他 -> is_reference=false
```

建议新增继承方法：

```python
def _inherit_context_entities(
    self,
    *,
    current_bag: EntityBag,
    history_bag: EntityBag,
    reference: ContextReferenceResult,
) -> tuple[EntityBag, str | None]:
    ...
```

继承策略：

- 如果存在 pending clarification，则优先继承上一轮任务实体，并用当前轮实体填补缺失字段。
- 如果 `reference.is_reference == False`，不继承，返回空 `inherited_bag`。
- 如果当前轮已有某实体类型，不继承同类型历史实体。
- 如果历史同类型只有一个高置信值，可继承。
- 如果历史同类型有多个候选：
  - 用户通过“第一个/第二个/上一个/前一个”等明确定位时，可以选择对应 frame。
  - 无法定位时，返回 `clarification_question`。
- 如果 reference 为弱追问且历史没有任何可继承实体，则澄清。

## 请求帧建议

短期内可以不引入持久化 frame 表，先基于 `recent_messages` 和 `EntityBag` 实现。

但实现时要为后续 frame 化留接口：

```text

  role
  entities
  query_preview
```

未来如果历史定位能力不足，再升级为真正的 frame 管理。

## 验收标准

- `.env` 未配置 LLM 时，规则兜底仍能正确处理多轮实体继承边界。
- LLM query rewrite prompt 明确禁止无条件继承历史实体。
- 当前轮强锚点默认开启新请求帧。
- pending clarification 优先级高于新请求帧。
- 用户补参场景能继承上一轮任务实体。
- 普通追问场景和澄清补参场景不会混淆。
- 明确上下文引用时才继承历史实体。
- 多候选只在需要继承且无法定位时澄清。
- `IntentRecognitionNode` 不新增历史实体继承职责。
